# N5.1 - Target Resistance (completing N5.1's blockade-power gap)

**Status:** Recorded. Closes the one remaining N5A gap N5.1's original evidence doc named explicitly: `province_blockade_bp()` returning raw attacker power with no resistance term. Full contested-zone rules (opposing fleets reducing each other's contribution), reverse indexes, and threshold events remain open, as N5.1 originally left them.
**Satisfies:** the target-resistance portion of [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N5.1, and the remainder of the N5A work packet in [05 - N5 Strategic Effects](../05_N5_STRATEGIC_EFFECTS.md)
**Scope:** a `_required_power()` function that gives every province a real, data-derived defensive rating, and a `province_blockade_bp()` that expresses attacker power as a bp fraction of that rating instead of returning the raw sum. No new persistent state, no defending-fleet-presence modelling (that is Contested Zones, a distinct still-open mechanic), no siege/economy/UI consumer changes beyond the two call sites (`EconomySystem`, `WarfareSystem`) N5.2 already wired to `province_blockade_bp()`.

## Why this was worth doing before moving to siege/repair/trade

Both of N5.2's evidence docs already flagged the same concrete problem: `province_blockade_bp()` summed raw fleet `total_blockade_power` (1 per war galley in the placeholder ship data) and treated that sum directly as a basis-point percentage. For any realistic fleet size this produced a near-zero blockade level - the N5.2 evidence doc's own worked example showed a 5-galley combined fleet registering as "5 bp," i.e. 0.05% blockaded, out of a possible 10000. The ledger and war-score consumers built on top of that number in N5.2 were consequently exercising formulas correctly but against an input that could barely move in practice. Leaving that as-is while building coastal siege integration, repair/port effects, and a trade-protection hook on the same query would have meant every subsequent consumer inherited the same "technically wired, practically inert" problem. Fixing the shared query once, before adding more consumers to it, is the smaller and more valuable packet.

## Architectural choices

**Required power reuses two fields that already exist on real data, not new authored content.** 05_N5 lists five possible resistance factors; this slice picks the two that already have real per-province/per-port values sitting unused for this purpose: `economy.base_tax`/`economy.base_production` (present on every province via `EconomySystem._make_province_economy()`, N5.2's own ledger integration already treats their *modified* form as "how much this province is worth") and `NavalDefinitions`' per-port `harbour_level` (present on every registered port since N1.1's data audit, previously consumed only by `ConstructShipCommand`'s `required_harbour_level` gate). Neither required a new JSON field, a new building type, or new fixture authoring - both were sitting in already-reviewed data.

**Development uses the raw `base_tax + base_production` values, not the fully modified `province_outputs()` figure.** `province_outputs()` needs `EconomyDefinitions` (trade-good prices, building modifiers, terrain tables) to compute, and `economy_system.gd` already `preload()`s `blockade_system.gd` for the N5.2 ledger integration - calling back into `EconomySystemScript.province_outputs()` from `BlockadeSystem` would create a preload cycle between the two scripts. Using the two raw economy fields directly avoids the cycle entirely and keeps the formula self-contained within `BlockadeSystem`, at the cost of not reflecting terrain/building/control modifiers in the resistance number the way the ledger's own loss figure does. This is a deliberate simplicity/cycle-avoidance trade-off, not an oversight.

**A land "fort" concept already exists in `WarfareSystem._fort_level()` but was not reused.** Land siege pacing already has a fort-level modifier (`warfare_system.gd` lines ~515-517), which looked at first like exactly "harbour/fort/shipyard level" from 05_N5's own wording. It was not reused because it is presently dead code - `"fort"` is not a definable building anywhere in `assets/economy_definitions.json`'s `buildings` dict, so `_fort_level()` can never return anything but 0 for any province today. Wiring blockade resistance to an always-zero value would have been a resistance term in name only. `harbour_level` was chosen instead because it is real, populated, per-port data (every fixture port has `harbour_level: 1` today) that already varies meaningfully.

**Defending fleet presence is explicitly not part of this slice.** 05_N5 lists it as one of several resistance factors, but a friendly fleet contesting a blockade is a *dynamic, per-tick* fact about fleet positioning, not a fixed property of the province the way development or harbour level are - it is squarely 05_N5's own separate "Contested Zones" concept ("opposing eligible fleets reduce or eliminate blockade contribution"), which N5.1's original evidence doc already named as its own open item. Folding it into `_required_power()` would have blurred two genuinely distinct mechanics (a province's static defensive rating vs. a dynamic naval contest) into one function. It remains open, unchanged from N5.1's original scope.

**`_side_blockades_other()` (N5.2's war-score helper) was updated to require a genuinely positive, resistance-adjusted `province_blockade_bp()` rather than mere candidate presence in `blockaded_provinces_for_fleet()`.** Before this change, a fleet within range of a hostile coastal province counted as "blockading" for war-score purposes regardless of whether its power actually exceeded that province's required power - inconsistent with adding a resistance term at all. A weak fleet that cannot overcome a well-defended port's resistance now correctly fails to register as an active blockade for war-score accumulation either, not just for the ledger.

## What was built

- `scripts/simulation/blockade_system.gd`: `NavalDefinitionsScript` preload; `BASE_REQUIRED_POWER`/`HARBOUR_LEVEL_REQUIRED_POWER` constants; `_required_power()`; `province_blockade_bp()` rewritten to express summed attacker power as a bp fraction of `_required_power()`, clamped to `[0, 10000]`, returning `0` early when attacker power is `0` (avoiding a needless `NavalDefinitions` load); `_side_blockades_other()` updated to check `province_blockade_bp() > 0` instead of raw candidate-list membership.
- `tests/naval_blockade_test.gd`: a `_expected_required_power()` test helper computing the same formula from real fixture data (not a hardcoded constant, so the assertions stay correct if the fixture's economy or port data changes); updated the "combining multiple fleets" assertions to the resistance-adjusted bp value; a new section proving the formula is neither always-zero nor always-full (5 raw power under-blockades a developed port; a fleet with exactly one ship per required-power point clamps to exactly 10000 bp, not past it); updated the economy-ledger section's expected-loss derivation to use the new bp value.

## Results (verified via `naval_blockade_test.gd`, exit 0, no errors)

- Picardie (base_tax 6, base_production 6, harbour_level 1) has a required power of 22 under this slice's formula; a combined 5-power attacking fleet registers as 2272 bp (~22.7%) - a real, informative percentage in place of the previous "5 bp" (0.05%).
- A fleet whose total power exactly equals the target's required power clamps to exactly 10000 bp (full blockade), not beyond it, for a fleet sized at exactly one ship per required-power point.
- The economy ledger's `blockade_loss` line, still verified against the live formula rather than a hardcoded number, now produces a materially non-zero loss for the same fixture that previously rounded to near-zero under the old raw-power formula.
- The war blockade score's day-by-day accumulation (still exercised over the same 48-simulated-day span N5.2 established) is unaffected in shape - the fixture's 3-power single-fleet blockade still exceeds Picardie's resistance and still registers as an active, accumulating blockade.
- No regression: re-ran all 42 Godot phase/core/naval tests after this round's changes - 41/42 pass, the one failure being the same pre-existing `country_label_layer_test.gd` frame-timing flake recorded in every N2/N3/N4/N5 evidence doc so far.

## Deliberately simple / deferred

- **No defending-fleet-presence term.** Still entirely open, as part of the separate Contested Zones mechanic - see the architectural note above.
- **No country/technology/building modifier hooks, no occupation/controller-state term.** 05_N5 lists both as possible resistance factors; neither is attempted. Occupation state in particular is a natural next addition (an occupied enemy province is arguably easier, not harder, to blockade) but is left for whichever slice takes up Contested Zones or siege integration, since it interacts with land-warfare occupation state this slice does not touch.
- **Required-power constants (`BASE_REQUIRED_POWER = 5`, `HARBOUR_LEVEL_REQUIRED_POWER = 5`, development weighted 1:1 with raw base_tax+base_production) are placeholder magnitudes, not approved balance.** Chosen only so that the fixture's small placeholder fleets produce a non-degenerate (neither always-zero nor always-full) result - the same "simplest explicit formula first" precedent every other first-slice formula in this codebase has used.
- **Development ignores terrain/building/control modifiers** for the preload-cycle reason given above - a province's resistance and its ledger-visible economic output can now diverge slightly (resistance uses raw base values; `blockade_loss` uses the fully modified `province_outputs()` value). Both numbers are internally consistent with themselves, but a building or terrain modifier changes one without changing the other.
- **No naval-blockade-scale stress/performance test.** `_required_power()` adds one more `NavalDefinitions.load_default()` disk read per non-zero `province_blockade_bp()` call, on top of the already-deferred "full-coast calculation meets approved budget" item from N5.1's original evidence doc - still open, now with a slightly larger per-call cost to account for.
