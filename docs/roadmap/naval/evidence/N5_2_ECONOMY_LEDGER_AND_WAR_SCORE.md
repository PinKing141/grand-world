# N5.2 - Economy Ledger Penalty and War Blockade Score (partial)

**Status:** Recorded for the economy ledger line and the war blockade score accumulator only. Coastal siege contribution, repair/construction/port effects, peace lifecycle, and the trade-protection hook remain open.
**Satisfies:** the economy-ledger-penalty and war-blockade-score portions of [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N5.2, and part of the N5D/N5B work packets in [05 - N5 Strategic Effects](../05_N5_STRATEGIC_EFFECTS.md)
**Scope:** wiring N5.1's existing `BlockadeSystem` query layer into two consumers that already existed - `EconomySystem`'s ledger and `WarfareSystem`'s war score - without adding any new persistent blockade state of its own. No siege, repair, port-status, sailor-recovery, or trade-protection effects are attempted here.

## Why this is two small wirings, not one new system

N5.1's evidence doc named this round's shape in advance: `BlockadeSystem` is a pure query, deliberately with nothing to accumulate against. Threshold events and war score needed *something* to persist a previous value - for war score that something already existed (`war["battle_score_attacker"]`/`occupation_score_attacker`/`ticking_score_attacker`, all fields on the pre-existing `war_registry` dict), and for the ledger it already existed too (`runtime["ledger"]`, computed wholesale by `EconomySystem.recalculate_all()`/`recalculate_country()`). Neither consumer needed a new registry - both needed one new field on state that already exists, and one call into `BlockadeSystem`'s existing query functions per tick. This is the same "extend, don't parallel" precedent N2.2 established for `navy_maintenance`, applied twice more.

## Architectural choices

**The ledger's `blockade_loss` is computed from each blockaded province's raw (pre-country-modifier) tax+production, in both `recalculate_all()` and `recalculate_country()` identically.** Country-wide tax/production modifiers (`_combined_country_modifiers()`) are applied *after* the province loop in both functions; `blockade_loss` is accumulated *inside* that same loop, before modifiers apply. This is a deliberate, simple choice consistent between both entry points - not "net of modifiers," but "gross loss at the province's raw output," matching the precedent that both ledger-computing functions must never diverge from each other (`EconomySystem`'s single-source-of-truth rule already applies to every other ledger line; `blockade_loss` follows it identically). `recalculate_country()`'s per-country province set is small enough that the N5.1-established "precompute `all_blockaded_provinces()` once" optimisation from `recalculate_all()` is not needed there - it queries `province_blockade_bp()` directly per owned province, since a single country's provinces are already a small, bounded set.

**`province_blockade_bp()`'s output is treated as basis points for the ledger formula exactly as N5.1 left it, with the same known limitation N5.1's evidence doc already flagged.** N5.1 explicitly deferred a target-resistance/normalisation term, meaning `province_blockade_bp()` today sums raw fleet `total_blockade_power` (currently 1 per war galley in the placeholder ship data) rather than a true 0-10000 percentage of blockade achieved. Feeding that number into `(tax + production) * blockade_bp / 10000` is dimensionally consistent with every other bp-scaled modifier in this codebase, but for realistic fleet sizes it currently produces a very small, often-zero economic effect - a handful of ships yields single-digit basis points out of 10000. This is not a new problem introduced by this round; it is N5.1's already-documented "closer to attacking power present than net blockade achieved" gap surfacing in a second consumer. Fixing it (normalising against a real resistance/required-power denominator) is still N5.1's open item, not repeated or worked around here.

**War blockade score is a bounded daily accumulator, not a fresh recompute-from-scratch like `occupation_score_attacker`.** `_update_war_scores()` already has two different shapes side by side: `occupation_score_attacker` is recomputed fully from current state every call (clamped, no memory of yesterday), while `battle_score_attacker` accumulates incrementally and only changes when a battle actually resolves. 05_N5's "Blockade War Score" requirements ("cannot grow... without an active eligible blockade," "releasing a blockade updates score according to approved retention/decay policy") describe the second shape, not the first - a fresh recompute has no way to "decay on release," it would just snap to whatever the current state implies. `BlockadeSystem.update_war_blockade_score()` therefore holds a running total on the war record and moves it by at most one point per day: +1 while one side holds an uncontested blockade advantage (blockading the other side and not simultaneously being blockaded back), -1 toward zero when neither side currently has an active blockade, held steady if both sides are blockading each other simultaneously (the simplest explicit rule for an otherwise-ambiguous contested case, deferring 05_N5's fuller "opposing eligible fleets reduce or eliminate blockade contribution" rule to the still-open Contested Zones item). The ±25 bound mirrors `ticking_score_attacker`'s envelope, not `occupation_score_attacker`'s ±50 - both are chosen as placeholder magnitudes, not approved balance numbers.

**This runs daily, inside `WarfareSystem._update_war_scores()`, not as a separate monthly `simulation_controller.gd` hook.** `_update_war_scores()` is already a daily system (`WarfareSystem.advance_day()` calls it every tick, itself already wired as a `daily_systems` entry). Adding a second scheduler registration and a second cadence for one more war-score component would create two different timing models for fields that all live on the same war record and feed the same `total_war_score` clamp - a needless inconsistency. 05_N5's own text ("bounded daily/monthly accumulation") permits either; daily was chosen because it is the cadence every sibling field already uses.

**`blockade_score_attacker` is added to both war-creation commands (`DeclareWarCommand`, `DeclareClaimWarCommand`) as an explicit `0` default, mirroring `battle_score_attacker`/`occupation_score_attacker`/`ticking_score_attacker`.** Every read of the field elsewhere still uses `.get("blockade_score_attacker", 0)`, so a save from before this field existed loads safely without a schema bump - the same backward-compatible pattern the other three score fields already rely on. No `SAVE_SCHEMA_VERSION` migration was needed for this reason.

## What was built

- `scripts/simulation/economy_system.gd`: `_empty_ledger()` gains a `blockade_loss` key; `recalculate_all()` precomputes `BlockadeSystem.all_blockaded_provinces()` once and accumulates `blockade_loss` per blockaded province inside its existing province loop; `recalculate_country()` gets the identical treatment inline (no precompute needed, per the architectural note above); both functions' `total_income` lines now subtract `blockade_loss`.
- `scripts/simulation/blockade_system.gd`: `update_war_blockade_score()` and its `_side_blockades_other()` helper - the only new functions this round adds to the existing pure-query class.
- `scripts/simulation/warfare_system.gd`: `_update_war_scores()` now calls `BlockadeSystemScript.update_war_blockade_score()` and includes `blockade_score_attacker` in the `total_war_score` clamp.
- `scripts/simulation/commands/declare_war_command.gd`, `declare_claim_war_command.gd`: new wars now initialise `blockade_score_attacker: 0` alongside the other three score fields.
- `tests/naval_blockade_test.gd`: extended (no new file) with an economy-ledger section and a war-blockade-score section.

## Results (verified via `naval_blockade_test.gd`, exit 0, no errors)

- `recalculate_all()`'s `blockade_loss` for the blockaded country equals `(tax + production) * blockade_bp / 10000` computed directly from `province_outputs()` and `province_blockade_bp()` - not a hardcoded fixture number, so the assertion holds regardless of the specific economic constants in `assets/economy_definitions.json`.
- `total_income` correctly subtracts `blockade_loss`.
- A country with no blockaded provinces shows exactly zero `blockade_loss`.
- `recalculate_country()` produces the identical `blockade_loss` value `recalculate_all()` does for the same country and state - the two ledger entry points do not diverge.
- War blockade score starts at zero, accumulates exactly one point per day while an uncontested attacker blockade is active, clamps at its configured maximum (25) rather than growing unbounded, and decays exactly one point per day toward zero once the blockading fleet is reassigned off the blockade mission.
- `total_war_score` includes `blockade_score_attacker` in its sum.
- No regression: re-ran all 42 Godot phase/core/naval tests after this round's changes - 41/42 pass, the one failure being the same pre-existing `country_label_layer_test.gd` frame-timing flake recorded in every N2/N3/N4/N5 evidence doc so far. `phase_5_warfare_test.gd` (land war score) is unaffected by the `_update_war_scores()` change, confirming `blockade_score_attacker` folds into the existing total without disturbing the three pre-existing score components.

## Deliberately simple / deferred

- **No target-resistance normalisation.** `province_blockade_bp()` is unchanged from N5.1; feeding its raw-power-sum output directly into the ledger/war-score formulas means the economic and war-score effect of a small, realistic fleet is currently very small. This is N5.1's already-open item, not newly introduced or silently patched here.
- **No coastal siege contribution, repair/construction/port effects, or sailor-recovery reduction.** 05_N5's "Province and Port Effects" lists several other minimum effects beyond the ledger line; only the ledger line is built this round.
- **No peace lifecycle for the blockade score.** A war's `blockade_score_attacker` simply stops being touched once `_update_war_scores()` skips a non-`"active"` war (the same early-exit every other score field already relies on) - there is no explicit "peace freezes/decays remaining blockade score" rule, consistent with the pre-existing peace/extinction-cleanup gap N3.3/N4.3 already documented.
- **No contested-zone reduction between opposing blockading fleets.** Both sides blockading each other simultaneously holds the score steady rather than applying 05_N5's fuller "opposing eligible fleets reduce or eliminate blockade contribution" rule - still open, same as N5.1 left it.
- **No trade-protection hook, AI valuation, or UI/map feedback.** All of N5.3/N5E remain entirely open.
- **No blockade-scale stress/performance test.** The ledger and war-score integration reuse `province_blockade_bp()`/`all_blockaded_provinces()` as-is; N5.1's own deferred "full-coast calculation meets approved budget" test is still open, now with two more call sites depending on it.
