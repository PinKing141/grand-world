# N5.1 - Blockade Started/Ended Events

**Status:** Recorded for the "Blockade started/ended" signal only - the first item in 05_N5's "Events and Queries" list. "Province blockade level changed across meaningful thresholds," "Port fully blockaded/unblocked," and "Coastal siege support changed" are explicitly not built - see below.
**Satisfies:** the blockade-started/ended portion of [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N5.1's threshold-events item, and part of [05 - N5 Strategic Effects](../05_N5_STRATEGIC_EFFECTS.md) "Events and Queries"
**Scope:** one new piece of persisted state (`CampaignWorldState.blockaded_provinces`), one new daily system (`BlockadeSystem.process_day()`), and two new events (`blockade_started`, `blockade_ended`). `SAVE_SCHEMA_VERSION` 8 -> 9.

## Why a pure query layer needed exactly one piece of persisted state

Every earlier N5.1 evidence doc repeated the same reasoning: `BlockadeSystem` has no stored state because a blockade level is a fact computed fresh from current world state, not an object with a lifecycle. Detecting a *transition* - "started" or "ended" - is different in kind from computing a *level*: a transition is inherently a comparison between now and some earlier point, and a stateless function has nothing to compare against. This is the same gap N5.1's very first evidence doc named in advance ("a pure query has nothing to compare against... threshold events... are consequently N5.2's job, once there is a persistent value for them to attach to"). `blockaded_provinces` is the minimum persisted value that makes any transition detection possible at all - a set of "which provinces were genuinely blockaded as of the last daily check," nothing more (not a reverse index of contributing fleets, not a history, not a bp value - just enough to ask "was this true yesterday").

## Architectural choices

**`blockaded_provinces` records a boolean fact (genuinely blockaded, resistance-adjusted bp > 0), not a bp value or a contributor list.** 07_DATA_COMMAND_EVENT_SAVE_CONTRACTS.md's original N0 proposal suggested a "Port/province -> active blockade contributors" reverse index. This slice deliberately builds something smaller: enough to detect a *started/ended* transition, not enough to answer "which fleets are contributing to this blockade right now" from saved state alone (that question is still answered by re-querying `BlockadeSystem.province_blockade_bp()`/`blockaded_provinces_for_fleet()` live, which remains cheap enough for the query patterns this session has built so far). A fuller contributor-list reverse index remains open for whichever future round needs it (AI reasoning traces, a UI breakdown tooltip, etc.) - not built speculatively here.

**Detection compares against `province_blockade_bp() > 0` (the same resistance-adjusted test every other N5.2 consumer already uses), not `all_blockaded_provinces()`'s cheaper candidate-list membership.** `all_blockaded_provinces()` answers "which provinces does at least one fleet target," before target resistance or contested-zone elimination are applied - using it directly would fire `blockade_started` for a province under attack by a fleet too weak to actually achieve any real blockade level, which would misrepresent what "started" means everywhere else in this codebase. `process_day()` still uses `all_blockaded_provinces()` as the cheap pre-filter (mirroring `EconomySystem.recalculate_all()`'s own "precompute the candidate set once, then run the expensive per-province check only on candidates" pattern) before applying the real bp test.

**`process_day()` is registered as its own `daily_systems` entry, after `NavalCombatSystem`, not folded into an existing system's function.** Every consumer this session has built so far extends an existing system (`EconomySystem`, `WarfareSystem`, `FleetLogisticsSystem`) because each of those already owns a related concern the blockade effect naturally attaches to. Transition detection does not belong to any of them - it is squarely `BlockadeSystem`'s own concern, the same system that already owns `is_fleet_eligible()`/`province_blockade_bp()`. Placing it last in the daily tick order (after `NavalCombatSystem`) means the recorded state reflects the day's fully settled outcome - including the contested-zones elimination this session's previous round added, which itself depends on whether `NavalCombatSystem` has already started a battle that day.

**Only "started/ended" is built, not the other three listed signals.** "Province blockade level changed across meaningful thresholds" needs a *previous bp value*, not just a previous boolean, plus a hysteresis/threshold-bucket design to avoid daily noise near a boundary - a materially larger design question than a binary flag. "Port fully blockaded/unblocked" is a second, distinct boolean transition (bp reaching/leaving exactly 10000) that would reuse the same `blockaded_provinces`-style infrastructure this round built, cheaply, as a near-term follow-on. "Coastal siege support changed" was already named as open in the coastal-siege-assist evidence doc for the identical reason. Building all four in one round would have been a much larger packet than this session's established slice size.

## What was built

- `scripts/simulation/campaign_world_state.gd`: `blockaded_provinces: Dictionary` (new field, cleared in `initialize()`); `SAVE_SCHEMA_VERSION` 8 -> 9 with a schema-8 migration step (`blockaded_provinces = {}`); checksum line `"blockaded_provinces=%s" % _canonical_variant(blockaded_provinces)`; `to_save_dict`/`apply_save_dict` wiring; validation that every recorded key is a known province.
- `scripts/simulation/simulation_event_bus.gd`: `blockade_started(province_id: int)`, `blockade_ended(province_id: int)`.
- `scripts/simulation/blockade_system.gd`: `process_day()` - diffs today's genuinely-blockaded set against yesterday's, emits the appropriate event per transitioning province, then replaces the persisted set.
- `scripts/simulation/simulation_controller.gd`: `BlockadeSystemScript` preload; `BlockadeSystem.process_day` registered as a `daily_systems` entry after `NavalCombatSystem`'s.
- `tests/naval_blockade_test.gd`: extended (no new file) with an events section (no event before any fleet exists; exactly one `blockade_started` on the day a genuine blockade first forms; no repeat event on a following unchanged day; exactly one `blockade_ended` on the day the blockade is released) and a save/load section (an active blockade record round-trips with an identical checksum; a corrupted save referencing an unknown province in `blockaded_provinces` is rejected).

## A real fixture gap found and fixed while writing the save/load test

Every fleet fixture in this file up to this point placed blockading fleets AT_SEA at `STRAITS_OF_DOVER` (a sea zone), reusing the sea zone's own ID as both `location_id` and (via `_add_fleet`'s helper) `home_port_id`. This had never been exercised through an actual save/load round trip before - every prior section of this test file queries `BlockadeSystem` directly against live in-memory state. The moment this round's save/load test tried to round-trip such a fleet, `apply_save_dict` correctly rejected it twice in succession: first for an "unknown home port" (fixed by overriding `home_port_id` to a real, owned port, Calais, for the save-testing fleet specifically - a fleet's home port should be a real port regardless), then for an "unknown location" (fixed at the fixture level: `_make_world()` now registers `STRAITS_OF_DOVER` itself as an explicitly-tracked, unowned province in `province_states`, rather than leaving it entirely absent - since every blockading fleet in this file sits there, this is the more broadly correct fixture, not a one-off patch). This is the same class of gap N4's evidence doc already documented and fixed once for `naval_combat_test.gd`'s own fixtures, surfacing again here because this was the first time *this* file's fleets went through a save/load round trip.

## Results (verified via `naval_blockade_test.gd`, exit 0, no errors)

- No `blockade_started` event fires before any eligible fleet exists.
- The day a fleet's blockade first becomes genuine (non-zero, resistance-adjusted bp), exactly one `blockade_started` event fires for that province, and `world.blockaded_provinces` records it.
- A following day with the blockade still active fires no additional `blockade_started` event for the same province - the transition, not the ongoing state, is what is signalled.
- The day the blockading fleet's mission is changed away from blockade, exactly one `blockade_ended` event fires, and the province is removed from `world.blockaded_provinces`.
- A save taken mid-blockade round-trips through `to_save_dict`/`apply_save_dict` with an identical checksum, and the reloaded world still records the active blockade.
- A save whose `blockaded_provinces` references an unknown province is rejected before any state mutation.
- No regression: re-ran all 42 Godot phase/core/naval tests after this round's changes - 41/42 pass, the one failure being the same pre-existing `country_label_layer_test.gd` frame-timing flake recorded in every N2/N3/N4/N5 evidence doc so far. This round touched `campaign_world_state.gd`'s core save/load path and bumped the schema version; every other test with its own save/load coverage (`naval_hud_integration_smoke.gd`, `naval_transport_gate_test.gd`, `naval_combat_test.gd`, `phase_5_warfare_test.gd`, `simulation_frame_rate_determinism_test.gd`) passed unaffected, confirming the new field and migration step did not disturb any other registry's round trip.

## Deliberately simple / deferred

- **No "province blockade level changed across meaningful thresholds" event.** Needs a persisted previous *bp value* plus a hysteresis design, not just a boolean - a larger follow-on, not attempted here.
- **No "port fully blockaded/unblocked" event.** *(Built in the very next round - see [N5_1_PORT_FULLY_BLOCKADED_EVENTS.md](N5_1_PORT_FULLY_BLOCKADED_EVENTS.md), which reuses this round's `blockaded_provinces` infrastructure exactly as anticipated here.)*
- **No "coastal siege support changed" event.** Already named as open in the coastal-siege-assist evidence doc.
- **No contributor-list reverse index.** `blockaded_provinces` is a boolean fact set, not "which fleets are contributing" - that question is still answered by re-querying live state, not from anything saved.
- **No naval-blockade-scale stress/performance test.** `process_day()` adds one more full `all_blockaded_provinces()` + per-candidate `province_blockade_bp()` pass, once per day, on top of every other consumer already deferring the same "full-coast calculation meets approved budget" item.
