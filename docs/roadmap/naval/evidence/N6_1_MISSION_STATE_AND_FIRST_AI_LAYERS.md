# N6.1 - Mission State/Command and First Naval AI Layers

**Status:** N6.1 complete. Mission command/state machine, explainable traces, staggered schedules, strategic posture, force construction, fleet organisation (admiral assignment and task-fleet merging), the sea-zone threat query (backing evasion/blockade-duty tactical decisions), and atomic transport-objective planning with land-AI handoff are all done, each narrower than 06_N6's full description of the finished item - see each section below for exactly what's included and what's deferred. Fleet *splitting*, and mission assignment driven by port/war-goal proximity rather than just a fleet's current zone, remain the only genuinely unstarted pieces within N6.1's own scope.
**Satisfies:** the N6A work packet ("mission command/state machine and shared evaluation APIs") from [06 - N6 Naval AI and UX](../06_N6_AI_AND_UX.md), and the corresponding items in [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N6.1.
**Scope:** exactly what N6A's own work-packet description says - the mission command layer every later AI packet will assign through, and a first AI system built to the same deterministic, staggered, explainable shape `StrategicAISystem` already established for land, not the full posture/threat/transport-planning sophistication 06_N6 describes for the finished pillar.

## Why this is the right first slice

06_N6 lists seven work packets (N6A-G) and a large "AI Planning Layers" section covering strategic posture, force construction, operational allocation, tactical decisions, a sea-zone threat map, engagement safety, and atomic transport planning with land-AI handoff. Attempting all of it in one slice would violate this roadmap's own delivery rule ("work one small packet to test-backed completion before opening the next dependency-heavy packet") - the same discipline every N1-N5 packet in this roadmap already followed. N6A is explicitly named as the foundation: "mission command/state machine and shared evaluation APIs." Everything else in N6.1-N6.3 needs a fleet to actually be able to hold and act on a mission, and needs an AI system with somewhere to record and explain its decisions, before it can build threat-driven or transport-planning logic on top.

## What was built

### Mission command/state machine

- `SetFleetMissionCommand.VALID_MISSIONS` gained `return_to_port` and `repair`, alongside the pre-existing `idle`/`blockade`. Both are self-completing missions, per 06_N6's own "Mission parameters... cancellation, and completion conditions are saved."
- `FleetMissionSystem` (new), registered as a `start_of_day_system` after `TransportSystem`: drives the two missions' actual behaviour, kept deliberately separate from `SetFleetMissionCommand` (which only validates/tags) and from `NavalAISystem` (which decides *when* to assign a mission, not what it does once assigned) - the same validate/apply/react three-way split N2-N5's command architecture already uses throughout.
  - `repair`: watches for full hull (`aggregate.total_hull >= aggregate.total_maximum_hull`) and clears to `idle`. Repair itself already happens unconditionally for any docked fleet at a legal repair port (`FleetLogisticsSystem._process_repair()`, unrelated to mission) - this mission's only job is completion-detection, matching 06_N6's "remain docked until repair threshold is met."
  - `return_to_port`: reuses the exact bounded-recovery shape `TransportSystem._attempt_recovery()` (N3.3) and `NavalCombatSystem._begin_retreat()` (N4.3) already established - try the current location if already legal, else `MaritimeGraph.nearest_matching()` for the nearest port the fleet's own country can legally dock at, issue the move, then clear to `idle` once actually docked. The third reuse of this exact pattern in this pillar, not a fourth slightly-different implementation.
- `patrol`, `intercept`, `protect_transport`, `protect_coast`, and `transport` are not added to `VALID_MISSIONS` - each needs the threat map (N6C) or transport-planning layer (N6D) this slice does not build. `trade_protection` has no underlying trade system to protect anything yet, the same reason 05_N5's own trade-protection hook remains open.

### NavalAISystem

Built to `StrategicAISystem`'s exact established architecture - deterministic utility scoring, per-country schedules staggered off the same `AIDefinitions` "slot" land AI already uses (`_due()`/`_next_due_day()`, byte-for-byte the same functions), persistent decision state in `country_runtime(tag)["naval_ai"]` (a new key, `StrategicAISystem`'s own `"ai"` key untouched), bounded decision history/counts/rejected-candidates, and a `debug_snapshot()` explainability view - not a new architecture invented for naval, reused wholesale because it already satisfies 06_N6's "Explainable AI Traces" section almost verbatim (country/day/category/action/score/reason/next-planning-day are all already exactly what `StrategicAISystem`'s shape produces).

- **Strategic posture** (`_review_posture()`, every 60 days): maritime-capable check (owns at least one structurally real port - `MaritimeGraph.is_port_province`/`is_port_enabled`, independent of a port's content-review confidence), peace/wartime posture, and a flat desired-ship-count budget (ports × 2 at war, × 1 at peace). Not the fuller threatened/invasion/recovery/expansion posture spectrum, and not a heavy/light/galley/transport desired mix - both need the threat map and force-construction nuance N6B/C were always going to add on top.
- **Force construction** (`_plan_construction()`, every 20 days): compares current + queued ships against the desired count, respects a treasury reserve mirroring `StrategicAISystem._plan_economy()`'s own shape, and builds at the highest-harbour-level owned port with no construction already queued there. Every ship built is a `war_galley` - the same "not final balance, establishes the shape" caveat N2.1's own representative ship data already carries; a desired class mix is not attempted.
- **Fleet organisation** (`_plan_organisation()`, every 15 days): assigns a free, qualified admiral (alive, employed, not already commanding an army or another fleet - mirroring `NavalHUD._populate_admiral_options()`'s own eligibility filter exactly) to an unled fleet with at least two ships, and now also consolidates separate idle fleets sharing a port into one task fleet (`_consider_fleet_merge()`, reusing `MergeFleetsCommand` - see its own section below). Fleet *splitting* and mission-based operational allocation are not attempted.
- **Tactical decisions** (`_plan_tactical()`, every 5 days): two real rules, no threat map behind either.
  - *Retreat*: for a fleet in an active battle, sums `aggregate.total_attack` across each side's living fleets (the same field `NavalCombatSystem`'s own damage formula already keys off) and requests retreat once badly outmatched (below half the opposing side's power) - `RequestFleetRetreatCommand.validate()` remains the actual authority (minimum round count, still-active battle, ownership); this only decides when it is worth trying.
  - *Repair/return*: an idle fleet that is damaged (below 80% hull) or unsupplied gets sent toward repair (if already docked at a legal repair port) or `return_to_port` (otherwise), letting `FleetMissionSystem`'s own state machine drive the rest.
  - *Evade/blockade* (added in a follow-on round to this same packet - see "Sea-zone threat query" below): an idle at-sea fleet either evades a zone it cannot safely cover, or takes up blockade duty in one that's safe and has a reachable war target.
  - Continuing or reinforcing mid-battle is not attempted.
- **Wiring**: `NavalAISystem.process_day()` registered in `simulation_controller.gd`'s existing `ai_hooks` array, alongside `CampaignGoalSystem`/`StrategicAISystem` - not a new scheduling mechanism.

### Sea-zone threat query

`NavalAISystem._zone_threat(world, tag, zone_id)` - a coarse, on-demand query (not a persisted/cached map): sums hostile fleets' `aggregate.total_attack` in the zone at full weight and in adjacent sea zones (`MaritimeGraph.sea_neighbor_ids()`) at half weight, for any country currently at war with the fleet's owner. Recomputed fresh per call rather than cached-and-invalidated, the same "scan and filter, no parallel index to drift out of sync" precedent `BlockadeSystem`'s own query layer already established for an analogous problem. Backs two new `_plan_tactical()` decisions, both gated to an idle, unassigned, at-sea fleet with no transport burden - the same restraint `_consider_repair_or_return()` already applies:

- **Evade**: if the current zone's threat overwhelms the fleet's own power (below 80% of threat), assign `return_to_port` - reusing `FleetMissionSystem`'s own routing rather than a second movement path.
- **Blockade duty**: otherwise, if the country is at war and the zone has a live target (`_zone_has_blockade_target()` - a hostile-owned coastal province among the zone's land neighbours, the same reciprocal relationship `BlockadeSystem.blockaded_provinces_for_fleet()` already uses, minus its eligibility gate), assign `blockade`.

06_N6's "Sea-Zone Threat Map" also asks for "recent battles and sightings" and "friendly support arrival time" - neither is modelled; both need persisted history or a broader search this slice does not build. Reinforcement, and moving an idle *docked* fleet out toward a distant blockade target, are also not attempted - only a fleet already at sea in a qualifying zone is assigned.

### Atomic transport-objective planning and land-AI handoff

`NavalAISystem._plan_transport()` reads land AI's own existing `country_runtime(tag)["ai"]["target_province_id"]` - `StrategicAISystem` already picks this for the active war and drives idle armies toward it every `MILITARY_INTERVAL`, it just has no notion that some targets need a ship rather than a march order. No new "objective" concept was invented, and "confirm strategic value" is inherited for free - land AI already decided the target is worth pursuing.

A real, non-obvious correction happened while building this: the original design assumed a hostile war-goal port could simply be the transport destination. It cannot. `NavalAccessPolicy.can_dock()` deliberately never grants docking rights merely from being at war (01_N1's own "sailing into a hostile harbour is not the same act as marching an army into hostile territory") - a fact this slice's own first test run surfaced immediately (`CreateTransportOperationCommand` rejected the hostile port with "Access... is denied," not a bug, the system working exactly as N1.3 already specified). `_find_legal_landing()` was written in response: land at the target directly only when it is actually legally dockable (unclaimed, allied, or the country's own); otherwise, land at the country's own nearest port that has a genuine land route onward to the target - a beachhead, not an assault landing. Once ashore, "land-AI handoff" needs no explicit handoff code at all: a disembarked army is simply `ARMY_STATUS_IDLE` again at a real province, exactly like any army that just finished marching, and `StrategicAISystem._issue_army_orders()` already scans every idle army generically - it will find the army at its new beachhead and march it the rest of the way on its own next tactical tick, land route and all.

Escort assignment, "acceptable threat" routing beyond bare legality, and monitoring an operation once under way are not attempted - `TransportSystem`'s own N3.3 failure-recovery already covers interruption/loss deterministically regardless of who created the operation.

### Fleet organisation: task-fleet merging

`NavalAISystem._consider_fleet_merge()` closes the "group compatible ships into task fleets" half of 06_N6's "Operational allocation" - the lowest-ID port (for determinism, not necessarily the most valuable one) with two or more of the country's own docked, organisable, transport-free fleets merges them into one via a real `MergeFleetsCommand`, the exact same command a player uses.

The transport exclusion is a deliberate, discovered guard, not an assumption: `MergeFleetsCommand.apply()` (existing code, unchanged) only moves ships between `fleet_registry` entries via `FleetSystem.move_ships()` - it never touches a source fleet's `transport_operation_ids`. Merging a fleet mid-transport would silently strand that operation's own `fleet_id` reference, an orphaning bug in the same family as several this session's naval work has already found and fixed elsewhere (`_validate_transport_data` would eventually reject the resulting save). Rather than fix `MergeFleetsCommand` itself for a case naval AI need never trigger, `_consider_fleet_merge()` simply never proposes merging a transport-carrying fleet - the correct scope boundary, since the command's existing behaviour is fine for every case a *player* would actually use it for (nobody merges a fleet they're mid-voyage with either).

Fleet *splitting* - dividing a fleet for a smaller mission - is not attempted; nothing in this slice yet needs to detach part of a fleet rather than combine whole ones.

## Deliberately not built

- **Fleet splitting and port/war-goal-proximity mission assignment** - the remaining half of "operational allocation." Task-fleet merging and threat-gated blockade-duty assignment exist; splitting a fleet for a smaller mission, and assigning missions by port or war-goal proximity rather than just a fleet's current zone, do not.
- **Engagement Safety's full model** - class matchup, morale, admiral skill, positioning estimate, reinforcement arrival, retreat-destination availability, and objective value all feeding a conservative effective-power estimate. This slice's retreat/evade rules use raw summed `total_attack` only.
- **`patrol`/`intercept`/`protect_transport`/`protect_coast`/`transport`/`trade_protection` missions** - see "Mission command/state machine" above for exactly why each is blocked.

## Results (verified via `fleet_mission_system_test.gd`, `naval_ai_test.gd`, `naval_ai_threat_test.gd`, `naval_ai_transport_test.gd`, and `naval_ai_organisation_test.gd`, exit 0, no errors)

- **Repair mission**: a fully healed fleet on the `repair` mission clears to `idle` in a single `process_day()` call; a still-damaged one is left alone, and only the healed fleet's `fleet_mission_changed` event fires.
- **Return-to-port mission**: a fleet already docked at a legal port clears immediately with no move order; a fleet stranded at sea gets routed to its own nearest legal port via real `FleetMovementSystem` ticking, then clears once actually docked there.
- **Mission command legality**: both new missions pass `SetFleetMissionCommand.validate()`, exactly like `idle`/`blockade` already did.
- **Naval AI, against the real Iberian fixture** (the same one `phase_6_ai_test.gd` uses, not a bespoke one - 06_N6's own "AI Content Rollout" step 2 is "Portugal, Castile, and Aragon integrated into the existing Iberian slice"): Castile, Aragon, and Portugal each own at least one of the 29 real N0.3-reviewed fixture ports, and each built at least one ship across 215 simulated days, with a populated, inspectable decision history, a recognised posture, a positive desired-ship-count, and at least one construction-category decision. Granada owns only structurally-real-but-harbour-level-0 ports ("candidate" content-review confidence, not "placeholder-reviewed") - it is correctly judged maritime-capable, takes real posture decisions, and records a real, explained construction rejection (harbour too small), but never completes a ship - proving the harbour-level gate in `ConstructShipCommand.validate()` is what stops it, not the AI silently doing nothing. Navarre owns no port at all (genuinely landlocked, both historically and in the baked map data) and takes zero naval AI decisions - the maritime-capability gate in `process_day()` actually gates entry, not merely influences scoring.
- **Command discipline**: `world.global_counters["naval_ai_commands_submitted"]` is nonzero and every submitted command passed real `validate()` before `scheduler.submit()` - the AI never mutates naval state directly, the same contract `StrategicAISystem` already holds itself to.
- **Determinism**: an identical campaign seed run twice produces an identical ship count per country and an identical final checksum.
- **Threat-driven evasion**: a lone England fleet directly co-located with an overwhelming Burgundian fleet registers real, nonzero threat and evades toward port (`return_to_port`), with the decision's own recorded reason naming the threat and own-power values compared.
- **Threat-driven blockade assignment**: an equal-strength lone England fleet at the same location, with no hostile fleet anywhere nearby (zero threat) but at war with Burgundy (which owns the adjacent coastal province), is assigned `blockade` duty instead of sitting idle.
- **Peacetime control**: the identical safe, alone fleet with no war at all is correctly left `idle` - there is nothing to blockade, and the AI does not invent a target.
- **Transport-objective beachhead handoff**: an idle England army at Kent (confirmed to have no land or strait route to either Calais or Picardie - a genuine sea gap, not an assumed one) with land AI's own target set to Picardie (a hostile Burgundian port) and a transport-capable fleet docked at Kent is ferried not to Picardie itself (confirmed illegal to dock at) but to Calais - England's own port, confirmed to have a real land route onward to Picardie - through a real `CreateTransportOperationCommand`, with the army correctly `ARMY_STATUS_EMBARKING` and the decision recorded under the `transport` category.
- **Transport-objective controls**: no land objective at all produces no transport operation; an objective already reachable directly by land from the army's own province is correctly left for land AI's own march order, not ferried.
- **Fleet merge**: two separate, unled, single-ship fleets docked at the same port consolidate into exactly one (the lowest-sorted fleet ID survives with both ships; the emptied source fleet is removed from the registry entirely, not left as a ghost entry), with the decision correctly recorded as a real `MergeFleetsCommand`.
- **Fleet merge controls**: fleets docked at different ports never merge; a fleet carrying a transport operation is never merged away even when another idle fleet shares its exact port.
- No regression: re-ran the full naval/AI/warfare/character/core regression set (25 tests) plus `naval_hud_integration_smoke.gd`, `campaign_interface_shell_smoke.gd`, and `phase_6_integration_smoke.gd` (all full-scene, the last specifically exercising AI-state save/load) after wiring the new systems into `simulation_controller.gd`/`campaign_interface_shell.gd` - all still pass, confirming the new `country_runtime(tag)["naval_ai"]` key round-trips through save/load without a schema bump (a free-form runtime-values bag, the same precedent `"ai"` itself already set), that `FleetMissionSystem` sitting after `TransportSystem` in the start-of-day order doesn't disturb transport's own daily sweep, and that the new Naval tab/Sailors resource label don't break the shell's own outliner/alert assertions.

## Bugs found and fixed while building the tests (test fixtures, not implementation)

- The first `naval_ai_test.gd` draft assumed Granada owned no naval port at all and expected every maritime country to have a *completed* ship within 90 days. Both assumptions were wrong: Granada's provinces are structurally real ports (just harbour-level 0, "candidate" review confidence) making it maritime-capable, and `war_galley` takes 150 construction days, so Castile - whose AI schedule slot (0) pushes its first `_due()` check to day 60, since `world.current_day` is never actually seen as 0 by the time `ai_hooks` run - can't complete a ship before day 210. Fixed by correcting the fixture's own expectations (Granada: maritime-capable but harbour-inadequate, real rejections, zero ships; simulated days extended to 215) rather than changing any implementation to fit a wrong assumption.
- `naval_ai_threat_test.gd`'s first draft called `NavalAISystem._plan_tactical()` directly (for precise, single-decision control instead of a full staggered day loop) but never followed it with `scheduler.process_commands()` - `_submit()` only queues a command via `scheduler.submit()`, exactly like every other command-submitting system in this codebase; `advance_one_day()` normally calls `process_commands()` for you. Calling the planning function directly without also draining the queue left the fleet's `mission` field unchanged even though the AI had "decided" and recorded the correct trace. Fixed by calling `scheduler.process_commands()` immediately after each direct `_plan_tactical()` call.
- `naval_ai_transport_test.gd`'s first draft assumed a hostile war-goal port could be transported to directly. It cannot - `CreateTransportOperationCommand.validate()` correctly rejected it with "Access to destination port ... is denied," which is `NavalAccessPolicy.can_dock()` working exactly as N1.3 already specifies, not a bug. This was the design assumption that led to `_find_legal_landing()`'s beachhead logic in the implementation itself, not just a test fixture correction - the clearest example in this packet of a test surfacing a real, pre-existing architectural constraint rather than a mistake in new code.
