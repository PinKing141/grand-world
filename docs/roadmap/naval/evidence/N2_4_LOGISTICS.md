# N2.4 - Fleet Logistics: Supply, Attrition, Repair, and Admirals

**Status:** Recorded  
**Satisfies:** [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N2.4, and the N2D (basing/supply)/N2E (attrition/repair/admirals) work packets in [02 - N2 Fleet Logistics](../02_N2_FLEET_LOGISTICS.md)  
**Scope:** supply status, monthly attrition for unsupplied fleets, in-port repair, and admiral assignment/exclusivity/lifecycle. No naval combat, blockade, or mission logic - those are N4/later.

## A production-wiring gap found and closed first

Before starting N2.4, a check of `simulation_controller.gd` (where every other recurring system - `ArmyMovementSystem`, `WarfareSystem`, `EconomySystem` - registers itself against the scheduler) turned up that N2.3's `FleetMovementSystem.advance_day()` was never added there. It only ran inside `naval_fleet_movement_test.gd`'s own hand-built scheduler. In the actual running game, fleets given a `MoveFleetCommand` would have accepted the order and then never moved - the daily tick that resolves legs simply never fired. This is now fixed: `FleetMovementSystemScript.advance_day` is registered as a `daily_systems` entry alongside `ArmyMovementSystemScript.advance_day`, and this slice's own `FleetLogisticsSystem.process_day`/`process_month` were registered from the start (`start_of_day_systems` and `monthly_systems` respectively) rather than repeating the mistake. Recorded here rather than silently folded into N2.3 because it was found and fixed during N2.4 work, not before.

## Architectural choices

**Supply reuses N1.3's `supply_range_query`, but the origin gate had to widen.** `NavalAccessPolicy.supply_range_query()` rejected any origin that wasn't a sea zone (`can_sail(zone_id)`), because at N1.3 time it only had sea-zone callers. A fleet's real location can just as easily be a port - docked, or mid-route having just entered one - so the gate now accepts `is_sea_zone(zone_id) or is_port_province(zone_id)`. No existing caller passed a port, so this is additive; the existing `naval_access_policy_test.gd` coverage (including the closed-water rejection case) still passes unchanged.

**A docked fleet at its own base needed a check before, not instead of, the range query.** `MaritimeGraph.nearest_matching()` deliberately excludes the origin node from matching against itself (see that function's doc comment - it exists to find the nearest *other* qualifying node, e.g. for `find_legal_route`'s port filtering). Calling `supply_range_query` directly from a fleet already docked at a basing-right port would therefore search *past* that port for a different one, which is wrong. `FleetLogisticsSystem.recompute_supply()` checks `NavalAccessPolicy.can_base()` at the fleet's own location first and only falls through to the range query when that fails - the same "check yourself before asking Dijkstra to look elsewhere" pattern the class's own doc comment now documents for future callers.

**Repair is gated by `can_base`, not by the `supplied` flag.** "Supplied" and "can repair here" are different bars: an at-sea fleet within range of a port days away is supplied but cannot repair there, since 02_N2 requires "a legal repair port and a docked fleet." `FleetLogisticsSystem._process_repair()` checks `location_status == DOCKED` and `can_base` directly at the fleet's own location rather than trusting the broader `supplied` bool.

**Repair spends the ship definition's own `repair_rate_bp`, not a new global constant.** `ship_definitions.json` already carried `repair_rate_bp` per ship (500-600 depending on family) from N2.1 - authored and validated by `ShipDefinitions._validate()` but never consumed by any system until this slice. Using it instead of inventing a flat rate keeps the "per-definition" data path this roadmap has followed since N2.1's ship stats, and means heavier/lighter ships already repair at different rates with no extra work.

**Admirals mirror commanders exactly, including where the exclusivity check lives.** `CharacterSystem.assign_admiral()` sits next to `assign_commander()` and performs the identical atomic swap (steal from a previous fleet if reassigned, update both sides of the relationship). `AssignAdmiralCommand.validate()` rejects a character already commanding an army; symmetrically, `AssignCommanderCommand.validate()` was extended to reject a character already commanding a fleet - "a character cannot command an army and a fleet simultaneously" (02_N2 "Admirals") is enforced from both entry points, not just the new one. `CharacterSystem.kill_character()` now clears `fleet.admiral_id` on death exactly as it already cleared `army.commander_id`. Save validation (`_validate_naval_data`, which gained a `characters` parameter) rejects a fleet whose admiral is unknown, dead, or non-reciprocal, mirroring the pre-existing army/commander check in `_validate_character_data`.

**No new save-schema version.** Every field this slice needed - `fleet.supplied`, `fleet.admiral_id`, `ship.hull_bp`, `ship.crew_bp`, `ship.repairing` - was already present in the N2.1 record shape (`make_fleet_record`/`make_ship_record`), forward-designed before this slice existed. The only genuinely new field, `character.admiral_fleet_id`, follows the same convention `commander_army_id` already established: every character-creation call site sets an explicit default, every read uses `.get(key, "")`, so old saves simply default to "no assignment" with no migration step required.

## What was built

- `scripts/simulation/fleet_logistics_system.gd` (new): `recompute_supply()`, `process_day()` (supply recompute + repair), `process_month()` (attrition).
- `NavalAccessPolicy.supply_range_query()` origin gate widened to accept ports.
- `CharacterSystem.assign_admiral()`, and `kill_character()` extended to clear `fleet.admiral_id`.
- `AssignCommanderCommand.validate()` extended with the fleet-exclusivity check.
- `scripts/simulation/commands/assign_admiral_command.gd` (new), mirroring `assign_commander_command.gd`.
- `CampaignWorldState`: `fleet.supply_reason` field added; `character.admiral_fleet_id` field added at every character-creation site; `_validate_naval_data()` gained an admiral-reference check (and a `characters` parameter to support it).
- Five new `SimulationEventBus` signals: `fleet_supply_changed`, `fleet_attrition_applied`, `fleet_repair_progressed`, `fleet_repair_completed`, `admiral_assigned`.
- `simulation_controller.gd`: registered `FleetMovementSystem.advance_day` (closing the gap above), `FleetLogisticsSystem.process_day`, and `FleetLogisticsSystem.process_month`.
- `tests/naval_fleet_logistics_test.gd` and `tests/naval_admiral_test.gd`, both registered in `tools/testing/run_all_tests.py`.

## Results (verified via both new tests, exit 0, no errors)

- `naval_fleet_logistics_test.gd`: a fleet docked at its own port is immediately supplied with no failure reason. Repair restores a damaged `war_galley` (hull 5000bp, crew 8000bp) at exactly its `repair_rate_bp` (500) per day for both hull and crew, deducting treasury for hull and sailors for crew each day, with the fleet aggregate's `total_hull` staying in sync every step; the ship reports `repairing=true` until both reach 10000bp (10 days, hull-bound) and is untouched thereafter. Insufficient treasury blocks hull repair for that ship that day without deducting anything (crew repair is gated independently). A fleet docked without basing rights does not repair even though it may still be counted "supplied" by range. Attrition reduces an unsupplied fleet's hull and crew by 500bp/month, never touches a supplied fleet in the same pass, and floors at 1000bp after repeated months rather than destroying the ship.
- `naval_admiral_test.gd`: rejections verified individually - unknown fleet, a country that doesn't own the fleet, unknown character, a character not employed by the issuing country, and a character under the adult age. The happy path assigns an admiral and updates both sides of the relationship. Exclusivity is verified in both directions: an admiral cannot be assigned as an army's commander, and an army's commander cannot be assigned as a fleet's admiral. Re-confirming the same admiral to the same fleet is accepted; assigning them to a second fleet while still assigned to the first is rejected. Killing the admiral clears `fleet.admiral_id`. A save with a fleet admiral reference pointing at a nonexistent character is rejected by `apply_save_dict`.
- No regression: re-ran all 35 Godot phase/core/naval tests (`country_registry_test` through this slice's two new tests) after this round of `character_system.gd`/`campaign_world_state.gd`/`naval_access_policy.gd`/`simulation_controller.gd`/`simulation_event_bus.gd` changes - 34/35 pass. The one failure is the same `country_label_layer_test.gd` frame-timing flake already recorded as pre-existing and naval-unrelated in [N2_3_ORGANISATION_AND_MOVEMENT.md](N2_3_ORGANISATION_AND_MOVEMENT.md).

## Deliberately simple / deferred

- `SUPPLY_RANGE_DAYS` (5), the attrition loss rates (500bp/month), and the floors (1000bp) are placeholder first-slice constants, not approved N0 budgets - the same status N2.1's ship stats and N2.2's sailor formula already carry.
- Attrition is fully deterministic (no RNG stream): 02_N2 permits variance "only where approved," and none has been approved yet. A named RNG stream can be added later without changing the interface.
- Repair cost consumes a fraction (`REPAIR_COST_FRACTION_BP`, 5%) of the ship's purchase cost to go from 0% to 100% hull, and the same per-bp rate applied to `sailor_cost` for crew - a simple, explainable placeholder formula, not balance content.
- No mission-level "refuse to start outside supply range" gate yet (02_N2 "Missions may refuse to start outside range") - there are no missions to gate in this slice.
- No AI-driven admiral assignment or naval AI of any kind - out of scope until N2F/later, matching N2.1's equivalent deferral for ship AI.
- Disabled-ship handling (`ship.disabled`) is modeled in the schema but nothing sets it yet - hull never reaches 0 through attrition (floored at 1000bp) or any other system built so far; that trigger belongs to N4 combat.
