# N3.2 - Transport State Machine

**Status:** Recorded, except battle pause (blocked on N4 combat concepts not existing yet)  
**Satisfies:** [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N3.2, and the N3B/N3C work packets in [03 - N3 Maritime Transport](../03_N3_MARITIME_TRANSPORT.md)  
**Scope:** embark timing, carrier sailing (driven by the existing `FleetMovementSystem`), and destination revalidation/disembark/land handoff. Battle pause is out of scope - N4 has not built interception/combat yet, so there is nothing to pause for. Loss/retreat/recovery is N3.3.

## Architectural choices

**Two army statuses, not one, because "locked" and "absent" are genuinely different facts.** N3.1 used a single `embarked` status for the (then-instantaneous) whole operation. Real embark timing needs both: while embarking, the army is locked (can't take new orders) but still land-present (03_N3: "Army remains on land but is movement-locked... unavailable for new... orders" - present tense, still there); only once embarking completes does it become genuinely absent ("Army is removed from land-presence queries"). Splitting this into `ARMY_STATUS_EMBARKING` (present, locked) and `ARMY_STATUS_EMBARKED` (absent, aboard) makes both facts independently queryable rather than overloading one status flag to mean two different things depending on context.

**Completion-day timing reuses `EconomySystem`'s pattern, not `FleetMovementSystem`'s.** N2.3 discovered that `FleetMovementSystem`'s `next_arrival_day` is checked *before* the scheduler's day increment, so a leg only resolves on the tick *after* the day it was due - a real, if non-obvious, timing model. `EconomySystem._complete_constructions()` instead checks `completion_day > current_day` from a `start_of_day_system`, which runs *after* the increment, giving the simpler "an operation submitted on day D with a K-day cost completes exactly when current_day reaches D+K" guarantee. `TransportSystem.process_day()` is registered as a `start_of_day_system` (after `FleetMovementSystem`'s `daily_systems` entry has already moved fleets that tick) specifically so embark and disembark timing follow the simpler economy-style rule, while still reading each day's *already-updated* fleet position when checking for arrival. Two different timing models coexisting in one codebase would be a hazard if undocumented; this doc and the code comments both name which pattern applies where.

**A fleet carrying a transport operation cannot receive independent movement orders.** 03_N3: "Fleet remains docked/adjacent and cannot depart independently" (Embarking) and implicitly throughout sailing, since the operation itself now owns the fleet's movement authority. `MoveFleetCommand.validate()` rejects any fleet with a non-empty `transport_operation_ids`. This does not block `TransportSystem`'s own internal call to `MoveFleetCommand.apply()` - that bypasses `validate()` entirely by construction (an internal system-driven mutation, not a player command going through the validation gate), the same pattern `EconomySystem`'s completion functions already use for army/building creation.

**Sailing routes from the fleet's *live* location, not the operation's recorded origin - a real bug caught before it shipped.** A fleet can carry multiple armies via distinct reservations (03_N3 "Capacity Model"), each potentially embarking on a different timer. The first draft of `_advance_embarking()` computed the sailing route from the operation's stored `origin_port_id`; if a second operation's embark timer expired *after* the fleet had already sailed away for a first operation, that stale origin would no longer be where the fleet actually was. Fixed by reading the fleet's current `location_id` at the moment each operation's embark completes, and by having an operation whose fleet isn't currently docked simply wait another day rather than route from a stale position. This is a known simplification for multi-destination convoys (see "Deliberately simple / deferred" below), not a crash risk - the fix specifically closes the corruption/mis-route risk, even though true convoy scheduling remains future work.

**Destination must now be a real port, not merely "coastal."** N3.1's `validate()` accepted any province with a sea neighbor, because nothing yet consumed the destination beyond bookkeeping. Now that `TransportSystem` actually sails the fleet there and disembarks at that exact node, the destination must be something `MaritimeGraph`/`NavalAccessPolicy` can route to and dock at - i.e. a real, enabled, currently-dockable port. This is a tightening of N3.1's own already-listed validation rule ("Destination has no legal disembark connection"), not a regression: N3.1 explicitly deferred the real disembark mechanism, and this is what "real" looks like now that it exists. Amphibious landing onto a non-port coastal province remains explicitly out of scope (see below).

## What was built

- `CampaignWorldState`: `ARMY_STATUS_EMBARKING`; `TRANSPORT_STATE_EMBARKING`/`SAILING`/`DISEMBARKING`; `make_transport_operation_record()` now takes an explicit `completion_day` and starts in `embarking`.
- `scripts/simulation/transport_system.gd`: `embark_days()` (bounded-integer formula - base 3 days, +1 day per 5 regiments, -1 day with a commander assigned, +2 days if any carried ship is damaged, floored at 1 day - all placeholder first-slice constants, not approved N0 budgets); `process_day()` and its three per-state handlers (`_advance_embarking`, `_advance_sailing`, `_advance_disembarking`), plus `_complete_operation()` and the `_fail_operation()` safety valve described above.
- `CreateTransportOperationCommand`: destination validation tightened to require a real port with a legal port-to-port route; `apply()` now computes real embark timing and starts the operation in `embarking`, not `embarked`.
- `CancelTransportOperationCommand`: now gated on `state == embarking` (was `embarked`) - cancelling a fleet that has genuinely sailed is out of scope until N3.3's recovery rules exist.
- `MoveFleetCommand`: rejects a fleet with any active `transport_operation_ids`.
- `simulation_controller.gd`: `TransportSystem.process_day` registered as a `start_of_day_system`, after `FleetLogisticsSystem`'s entry.
- `tests/naval_transport_operation_test.gd` rewritten to exercise the real state machine (previous version asserted instantaneous `embarked`, which N3.2 correctly no longer produces).

## Results (verified via `naval_transport_operation_test.gd`, exit 0, no errors)

- All of N3.1's rejection/capacity/ownership coverage still holds, with the destination check now correctly rejecting a non-port coastal province (found dynamically, not hardcoded) and a foreign port with no access relation.
- Creating an operation reaches exactly `embarking`: the army is locked but still land-present, capacity is reserved immediately (before embarking even finishes), and an independent `MoveFleetCommand` against the carrying fleet is rejected.
- An already-embarking army cannot be embarked again; cancelling while embarking is penalty-free (matches N3.1's guarantee, now scoped correctly to the embarking state only).
- Full Channel journey, driven by the real scheduler (`FleetMovementSystem` as a `daily_system`, `TransportSystem` as a `start_of_day_system` - the exact production wiring): one regiment, an undamaged fleet, no commander costs exactly the base 3 embark days; the embark timer then puts the army aboard (no longer land-present at Calais) and orders the fleet to sail; the fleet's real two-leg Calais→Straits of Dover→Kent crossing (the same fixture N2.3 already proved) resolves in three more ticks per `FleetMovementSystem`'s established check-before-increment timing; one more tick for the disembark delay lands the army at Kent, fully idle, unlocked, and land-present there - with the operation record gone and capacity fully released.
- `MoveArmyCommand` against a real (not instantaneous-fixture) embarked army still returns exactly `"The army is embarked."`.
- Save/load round trip on a mid-embarking operation reproduces an identical checksum; corrupted saves (dangling army reference, dangling operation reference) are rejected; schema-6 saves migrate cleanly.
- No regression: re-ran all 38 Godot phase/core/naval tests after this round of `campaign_world_state.gd`/`move_fleet_command.gd`/`simulation_controller.gd`/`simulation_event_bus.gd` changes - 37/38 pass, the one failure being the same pre-existing `country_label_layer_test.gd` frame-timing flake recorded in every N2/N3 evidence doc so far.

## Deliberately simple / deferred

- **No battle pause.** 03_N3's `battle_paused` state needs interception/combat concepts N4 has not built. N3.3's job once they exist.
- **No amphibious (non-port) landing.** Destination must be a real port on both ends of the journey; landing on an arbitrary coastal province without port infrastructure is a real feature the roadmap implies but this slice does not attempt.
- **Multi-destination convoys on one fleet are safe but not elegant.** A fleet carrying two operations with different destinations and different embark timers will have the second operation simply wait (potentially for a long time) until the fleet happens to redock somewhere, rather than being actively rescheduled or split. No stranding or corruption results, but it is not a satisfying convoy experience - a known simplification, not a bug.
- **A fleet blocked mid-sail (access revoked while underway) leaves its operation "sailing" indefinitely**, tracking wherever `FleetMovementSystem` leaves the fleet, with no active recovery. `FleetMovementSystem` itself already halts the fleet safely (no teleport, no deletion - N2.3), so nothing corrupts; but the transport operation itself does not yet react to that block. Real recovery (follow the retreating fleet, deterministic loss on destruction, "no legal recovery -> explicit destruction") is N3.3.
- **`_fail_operation()` is a narrow safety valve, not the full loss/recovery policy** 03_N3 describes. It only fires when the destination itself becomes illegal at a state transition, and always resolves the army onto land wherever the fleet currently is - safe, but not the richer bounded-recovery rule the roadmap eventually wants.
