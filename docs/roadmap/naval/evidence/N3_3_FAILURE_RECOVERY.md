# N3.3 - Transport Failure Recovery

**Status:** Recorded for every trigger reachable without N4 combat. Fleet retreat/destruction and peace/extinction cleanup, both originally recorded as blocked below, have since been closed - see "Fleet retreat/destruction, closed" below and [N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md](N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md) respectively. `battle_paused` remains open.
**Satisfies:** [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N3.3, and the N3D work packet in [03 - N3 Maritime Transport](../03_N3_MARITIME_TRANSPORT.md)  
**Scope:** capacity-shortfall losses (triggered by the already-existing `FleetLogisticsSystem` attrition, not combat) and recovery/destruction for a transport fleet `FleetMovementSystem` has already halted mid-route due to an access/ownership change. Combat-driven capacity loss for a fleet that survives, fleet retreat, and peace/extinction cleanup were all originally out of scope pending N4/other-slice dependencies; the latter two have since been closed (see below) once those dependencies landed. `battle_paused` remains open.

## Why this is a partial slice, and what's genuinely blocked vs. deferred

03_N3's "Loss and Recovery Policy" describes a unified system where combat damage, interception, fleet retreat, and destruction all feed into transport loss/recovery. None of that machinery exists yet - N4 (Combat) has not started. Rather than wait for all of N4 before touching N3.3 at all, this slice asked a narrower question: *which of 03_N3's failure triggers can already happen today, without combat?* Two can:

1. **Capacity dropping below what's reserved** - already possible today, because `FleetLogisticsSystem`'s monthly attrition (N2.4, unsupplied fleets) can push a ship's hull below `TransportSystem.DAMAGED_CAPACITY_THRESHOLD_BP`, which `usable_capacity()` already treats as zero contribution. This was true the moment N2.4 and N3.1 both existed; nothing in N3.3 needed to invent a new trigger, only a new response.
2. **A carrier fleet getting halted mid-route** - already possible today, because `FleetMovementSystem`'s per-leg access revalidation (N2.3) already halts a fleet safely (not deleted, not teleported) when the destination or an intermediate port stops being legally reachable. N3.3 only needed to teach `TransportSystem` to notice and react to a halt that already happens.

Everything else in 03_N3's policy needs a trigger this codebase cannot yet produce: there is no fleet "retreat" state-setter at all (`FLEET_LOCATION_RETREATING` exists as a constant but nothing ever assigns it), and there is no combat system to damage/sink/capture ships. Those are correctly N4's job, not something to fake here. Peace and country-extinction cleanup are a different kind of gap: `CountryDepthSystem._cleanup_extinct_country_references()` (found while scoping this slice) does not clean up armies, fleets, or commanders on country extinction today - a pre-existing gap this slice's transport operations simply inherit rather than introduce. Fixing that belongs to whichever slice first makes extinction cleanup real for armies/fleets in general, not to transport specifically.

## Architectural choices

**The shortfall check runs daily for every fleet with an active operation, not only after attrition.** `_resolve_capacity_shortfalls()` doesn't know or care *why* `usable_capacity` fell - it just compares it against `reserved_capacity` every day. This means the exact same code will correctly handle a future N4 trigger (combat-damaged ships) with zero changes, the same way `FleetLogisticsSystem`'s repair/attrition already didn't need to know *why* a ship was docked or unsupplied.

**Losses are applied in strict sorted operation-ID order, and each operation absorbs its own reservation before the deficit spills to the next.** 03_N3 asks for "deterministic regiment/strength losses using stable operation and army ordering" without prescribing exact math; sorted-ID order matches every other stable-ordering convention already established across N2/N3 (ship-ID order for repair allocation, fleet-ID order for aggregate recompute, etc.). Strength scales down proportionally with integer division (`strength_per_regiment = strength / regiment_count`), and an army reduced to zero regiments is destroyed outright rather than left as an inert zero-strength record - "no army may be... lost from all registries" cuts both ways: it must not vanish silently, but a genuinely wiped-out army also must not linger as a ghost entry.

**Recovery tries the original destination again before searching for a new one**, since the access loss that caused the halt might already be resolved by the time recovery runs (e.g. a captured port recaptured). Only if that still fails does it fall back to `MaritimeGraph.nearest_matching()` for the nearest port the country can still legally dock at - and that call needed a small fix of its own: `nearest_matching()` deliberately excludes the search's own origin from matching against itself (the same subtlety `FleetLogisticsSystem.recompute_supply()` already had to work around in N2.4), so a fleet already halted *at* a perfectly legal port has to be checked directly first, or recovery would incorrectly search *past* it for somewhere else.

**Detecting "the fleet has been blocked" reuses an existing invariant rather than adding a new flag.** Because `MoveFleetCommand` now rejects any independent order against a fleet carrying a transport operation (N3.2), the *only* way such a fleet can stop moving is either arriving at the operation's actual destination, or `FleetMovementSystem` halting it after a blocked leg. So "docked or at-sea, no remaining path, no destination" that *isn't* the operation's destination is an unambiguous signal of a block - no new state field needed on the fleet record.

## What was built

- `TransportSystem._resolve_capacity_shortfalls()` / `_apply_capacity_loss()`: the deterministic loss mechanism described above, called at the top of `process_day()`.
- `TransportSystem._attempt_recovery()`: bounded recovery (retry original destination, then nearest legally-dockable port), called from `_advance_sailing()` when a fleet is detected halted somewhere other than the operation's destination.
- `TransportSystem._destroy_stranded_operation()`: the explicit-destruction terminal path, used both when recovery finds no legal port and when a capacity loss reduces an army to zero regiments.
- Three new `SimulationEventBus` signals: `transport_capacity_shortfall`, `transport_operation_rerouted`, `transport_operation_army_lost`.
- `tests/naval_transport_recovery_test.gd` (new), registered in `tools/testing/run_all_tests.py`.

## Results (verified via `naval_transport_recovery_test.gd`, exit 0, no errors)

- **Capacity shortfall, partial then total.** Two armies (1200 and 800 regiments - deliberately large to exercise the placeholder `transport_capacity: 1000`-per-ship data meaningfully, the same "not final balance" caveat N2.1's ship stats already carry) fully reserve a two-ship fleet's 2000 capacity. Damaging one ship halves usable capacity to 1000, producing a 1000-unit deficit; the lower-sorted operation absorbs it first (1200 → 200 regiments, strength scaled proportionally), the second operation is untouched, and reserved capacity is trimmed to exactly match what survives. Damaging the second ship too (usable capacity → 0) wipes out the first army entirely (destroyed, not left at zero) and the remaining deficit continues into the second army, destroying it too - both operations and all fleet reverse-references cleanly removed.
- **Blocked-fleet recovery.** An England army embarks at Calais bound for Kent; mid-crossing, Burgundy captures Kent (both `owner` and `controller`, the exact gotcha N2.3 already documented for this identical fixture). `FleetMovementSystem` halts the fleet safely at the Straits of Dover. `TransportSystem` detects the halt, finds Kent still illegal, finds Calais (England's own port) as the nearest legal alternative, reroutes, and the fleet completes its journey back to Calais - the army lands there fully idle and unlocked, not stranded, duplicated, or destroyed.
- **Explicit destruction.** Verified directly against `_destroy_stranded_operation()` (constructing a genuine "no port anywhere on the real map is reachable" scenario is impractical - any province absent from a sparse test fixture's ownership map reads as "unclaimed," and an unclaimed port is dockable by anyone per `NavalAccessPolicy.can_dock`, so true full isolation needs the entire real map under hostile control, not a lightweight fixture). The army is fully erased from `army_registry` (not left at zero), the operation record is removed, and the fleet's reverse reference is cleared.
- No regression: re-ran all 39 Godot phase/core/naval tests after this round of `transport_system.gd`/`simulation_event_bus.gd` changes - 38/39 pass, the one failure being the same pre-existing `country_label_layer_test.gd` frame-timing flake recorded in every N2/N3 evidence doc so far.

## Bug found and fixed before shipping (test, not implementation)

`naval_transport_recovery_test.gd`'s blocked-fleet loop originally checked `String(world2.get_army("army_c").get("current_province_id", ""))` to detect completion. `current_province_id` is always an int once the key exists (which it always does), so `String()` on that int hit the exact "Invalid call 'String' constructor" pitfall this session has hit and documented multiple times before (`JSON.parse_string` and plain numeric Variants both throw under `String()`, not just `str()`). The type error aborted `_run()` silently mid-loop without ever reaching `quit()`, leaving the headless process hanging until the tool timeout killed it - a instructive reminder that an uncaught runtime type error in this harness doesn't fail loudly, it just hangs. Fixed by checking registry membership directly (`not world2.army_registry.has("army_c")`) instead of stringifying a field never needed as a string.

## Fleet retreat/destruction, closed

N4 built a real fleet-retreat mechanic after this doc was first written, closing the trigger that used to block this item outright. Checking what that mechanic actually produces turned up two distinct cases, only one of which needed new code:

**A carrier fleet retreating elsewhere needed nothing new.** `NavalCombatSystem._begin_retreat()` drives a retreating fleet through the exact same `FleetMovementSystem` a normally-ordered fleet uses (`FLEET_LOCATION_RETREATING` is handled identically to `FLEET_LOCATION_MOVING`), so once the retreat completes, the fleet settles at a real location exactly as if `MoveFleetCommand` had sent it there. `_advance_sailing()`'s existing "settled somewhere other than the operation's destination" check - built for the access/ownership-halt case above - can't tell a halted fleet from a retreated one, and doesn't need to: it already reroutes correctly either way.

**A carrier fleet destroyed outright was a real, separate gap.** Both `_finish_battle()` (a losing fleet reduced to zero ships) and `_begin_retreat()`'s "no legal retreat" branch erase the fleet from `fleet_registry` directly. `_advance_sailing()`'s settled-check only recognises `FLEET_LOCATION_DOCKED`/`FLEET_LOCATION_AT_SEA`; an erased fleet's `get_fleet()` returns `{}`, whose `location_status` reads as `""` - neither DOCKED nor AT_SEA - so the check silently never fired. The operation, and the embarked army riding inside it, would sit forever as a live-looking record pointing at nothing, and `_validate_transport_data` would reject the very next save (`"Transport operation references unknown fleet"`). Fixed with one guard at the top of `process_day()`'s per-operation loop: if the operation's fleet no longer exists in `fleet_registry` at all, treat it exactly like every other unrecoverable loss (`_destroy_stranded_operation()`, the same terminal path capacity-shortfall and no-legal-port already use) rather than dispatching to whichever state-handler was about to silently no-op.

While proving this, `_begin_retreat()`'s "no legal retreat" branch turned out to have its own bug: it erased the fleet record but left its surviving ships as orphaned entries in `ship_registry`, referencing a `fleet_id` that no longer existed - `_validate_naval_data`'s ship loop would have rejected that save too (`"Ship references unknown fleet"`). Fixed alongside, erasing the fleet's ships first, mirroring how `_finish_battle()`'s own destruction path (via `_remove_sunk_ships`) already keeps ships and fleets in lockstep.

Verified via `tests/naval_transport_combat_loss_test.gd`: rather than scripting a multi-round battle to a precise one-shot kill (impractical - gradual ship damage always crosses the pre-existing 50%-hull capacity threshold on its way to zero, which triggers the *already-working* capacity-shortfall sweep before a ship or fleet is ever fully gone; reproducing the "no legal retreat" branch specifically through real geography needs the entire map under hostile control, the same reachability limit already documented below for `_destroy_stranded_operation`'s own explicit-destruction case), the test drives the exact post-erasure state directly: a transport operation naming a fleet that was never added to `fleet_registry`, standing in for "combat already erased it." `process_day()` cleans it up, the embarked army is lost with it, `transport_operation_army_lost` reports a battle-specific reason, and the resulting save round-trips cleanly. Re-ran the full naval/warfare/character/core regression set afterward - no regressions from either fix.

## Deliberately blocked / deferred

- **Combat-driven capacity loss (sinking, disabling, capture) for a fleet that survives** - already covered; the shortfall mechanism built in this slice needed no changes once N4 existed to trigger it.
- **Interception and battle_paused** - holding an operation still mid-embark/sail while its own fleet is actively fighting, rather than only reacting after combat has already settled the fleet's fate one way or another - remains open, as already recorded in N3.2's evidence doc.
- ~~**Peace/country-extinction cleanup for transport operations**~~ - closed; see [N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md](N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md).
- **Recovery only reacts to a halt found during `_advance_sailing`'s daily check** - if a fleet is blocked and then immediately un-blocked before the next tick (a same-day flicker), recovery never fires and the operation simply continues sailing normally, which is the correct outcome, just worth naming as the boundary of what "detects a block" means here.
