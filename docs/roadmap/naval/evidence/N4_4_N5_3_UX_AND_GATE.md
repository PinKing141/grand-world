# N4.4 / N5.3 - Naval Combat and Blockade UX/Gate

**Status:** Complete for every mechanic the simulation actually implements. Positioning/morale/capture (N4.2's own still-open items) and peace-treaty mid-battle disengagement have nothing to surface yet, so this slice does not fabricate UI for them.
**Satisfies:** [10 - Delivery sequence and checklist](../10_DELIVERY_SEQUENCE_AND_CHECKLIST.md) N4.4 and N5.3, and the N4E/N5E work packets in [04 - N4 Naval Combat](../04_N4_NAVAL_COMBAT.md) and [05 - N5 Strategic Effects](../05_N5_STRATEGIC_EFFECTS.md).
**Scope:** the battle marker/panel/report/retreat control the Channel scenario needs to actually *play* a naval battle, and the port/zone/coast feedback, alerts, and ledger lines the same scenario needs to *see* a blockade - both riding entirely on N4/N5's already-complete backend, and both bounded by what that backend actually tracks.

## Why this waited, and why it didn't need to wait for N6

Every earlier N4/N5 evidence doc deferred its own UX slice "to a follow-up round" - correctly, since a panel showing fields the backend doesn't compute yet (morale, positioning, captures) would either lie or need placeholder data, and 04_N4/05_N5 both explicitly list those mechanics as separate, still-open work packets (N4B/C's remainder, N5E's trade-protection hook). This slice drew the line at exactly what N4A-D and N5A-D already produce: battle records with round/hull-lost/ships-sunk/winner_side, and a pure blockade query layer with bp/tier/required-power. Nothing here anticipates a mechanic that doesn't exist.

## What was built

### Battle marker (N4E)

`ConflictMarkerLayer` (`scripts/ui/conflict_marker_layer.gd`) gained a third marker family, `naval_battle`, structurally parallel to but separate from the existing `battle`/`siege` pair:

- Its own `MultiMeshInstance3D` (`_naval_battle_markers`), using the marker atlas's existing "navy" icon (`atlas_index: 1` - "An original navy marker in the generated marker atlas" was already listed as reusable foundation in this roadmap's own README, unused until now).
- Its own count/visibility/instance debug accessors (`debug_naval_battle_count()`, `debug_naval_battle_visible()`, `debug_naval_battle_instances()`) rather than folding into `debug_battle_count()`/`debug_draw_count()`/`debug_marker_instances()`/`debug_priority_order()` - those four already have exact-value assertions in `conflict_marker_layer_smoke.gd`/`conflict_marker_stress_smoke.gd` that predate naval battles entirely and must keep meaning exactly what they meant before.
- Naval battle records feed the *same* clustering/anchor/fade pipeline land markers use (`_cluster_records()`, `anchor_world_position()`) - a sea zone is just another `province_id` in the same `ProvinceGraph`, so nothing zone-specific was needed. `_clusters` (the shared click-detection array) gets battle, then siege, then naval-battle clusters appended in that fixed order every rebuild.
- Colour intensifies with total hull lost so far (blue toward red), the same "damage so far" signal siege markers already give via `progress_bp`.
- Clicking a naval-battle-typed marker now routes to `NavalHUD.select_battle()` instead of `WarHUD.focus_conflict_marker()` - a small branch in `_on_map_click_requested()` keyed on `marker_type`.

### Battle panel, report, retreat control (N4E)

`NavalHUD` (`scripts/ui/naval_hud.gd` / `scenes/ui/naval_hud.tscn`) gained:

- A battle list scoped to `_player_active_battles()` - active battles where any current fleet on either side is owned by the player, the same live-membership test the marker layer's click routing implicitly relies on agreeing with.
- A detail view for the selected battle: sea-zone name, round, and each side's fleet count/hull-lost/ships-sunk, straight from the battle record.
- A "final report" reading the *most recently completed* battle either side of a war the player belongs to fought - found via `DiplomacySystemScript.side_in_war()` against the battle's `war_id` rather than live fleet ownership, since a completed battle's fleets may since have been destroyed and erased entirely. This needed no new persisted state: completed battle records are already a permanent history snapshot (`_validate_naval_battle_data`'s own doc comment already says so), so reading `naval_battle_registry` directly is correct, not a workaround.
- A retreat button added to the existing fleet panel's action row (not a separate control tied to the battle list), reusing `RequestFleetRetreatCommand.validate()` directly for its enabled state - "earliest legal date" (04_N4 "Player Feedback") *is* the tooltip explaining why validate() rejected it (not yet the minimum round count, not in a battle at all, etc.), rather than a second parallel eligibility calculation.

### Port/zone/coast feedback, alerts, ledger (N5E)

- `NavalHUD` gained a blockade label for whichever province is currently selected (any coastal province, not only the player's own - seeing an enemy's blockade is just as relevant): tier name, bp as a percentage, and `BlockadeSystem.required_power()` alongside it. `required_power()` (previously `_required_power()`) was made public specifically for this - 05_N5's own "Query breakdown for UI, AI, economy, siege, and debug use" already asked for exactly this kind of read access, and nothing about the function was actually private to its two existing callers.
- A fleet mission control (`SetFleetMissionCommand`, `idle`/`blockade`) on the fleet panel, showing `BlockadeSystem.effective_power()` once a fleet is actually on blockade duty.
- A "show blockade map" button producing a coastal overlay (yellow at light blockade, red at full) via `MapHUD.set_strategy_map_overlay()` - the exact same mechanism `WarHUD`'s war/relations/access map buttons already use, not a new overlay system.
- `WarHUD`'s war summary line gained a blockade-score readout (`war["blockade_score_attacker"]`) alongside the total war score it already showed.
- `EconomyHUD`'s ledger gained `navy_maintenance` and `blockade_loss` line items. Both values were already folded into `total_income`/`total_expenses` since N2.2 and N5.2 respectively - this was a genuine display gap the roadmap's own "Economy ledger blockade-loss line" ask caught: the ledger *label* had simply never been updated to show either figure on its own line, land's `army_maintenance` was the only maintenance ever surfaced.
- `CampaignInterfaceShell` gained a BLOCKADES outliner section (the player's own blockaded coasts, tier/bp, click-to-focus-and-open-naval) and two new top-bar alerts - "Naval battles (N)" and "Blockaded ports (N)" - mirroring the existing FLEETS/transport-operations alert pattern exactly, including the same `_open_naval` action.
- Toast notifications (`NavalHUD._notify()`, the same mechanism transport events already use) fire for naval battle start/end, fleet retreat, fleet/ship loss, and every blockade threshold event - `blockade_started`/`ended`, `port_fully_blockaded`/`unblocked` - scoped to provinces the player owns, so a blockade happening to a third country elsewhere on the map does not spam the player's toast queue.

"War/economy explanation" (05_N5 "UI and Map Feedback") *is* the bp/tier/required-power breakdown itself in this slice - there is no positioning/morale-level detail underneath it yet to narrate further.

## Deliberately not built

- **Positioning breakdown, morale, commanders-as-a-roster, captures** - N4.2's own still-open items (`04_N4_NAVAL_COMBAT.md`'s "Deliberately simple / deferred"). The battle panel shows exactly the fields `NavalCombatSystem` actually computes; adding UI rows for mechanics that don't exist would either be permanently empty or silently fabricated.
- **Peace-treaty mid-battle disengagement.** A war reaching `status == "ended"` via a normal peace treaty does not currently stop an already-active naval battle (`NavalCombatSystem._resolve_battles()` has no war-status check, only a battle-status one) - a real, separate gap already flagged in this session's [N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md](N3_N4_N5_PEACE_EXTINCTION_CLEANUP.md), not something a UX slice should paper over with a control that would have nothing correct to do.
- **Trade-protection hook and its UI/alerts (N5E's other half).** No trade-network system exists yet for a naval blockade to protect or threaten; 05_N5 names this as its own separate item, not part of "UI and Map Feedback."

## Results (verified via `naval_hud_integration_smoke.gd`, `conflict_marker_layer_smoke.gd`, and `naval_battle_blockade_stress_smoke.gd`, exit 0, no errors)

- **Battle panel end-to-end.** `naval_hud_integration_smoke.gd` (extended) drives England's own constructed fleet through a real embark/transport cycle, then injects a synthetic hostile fleet and battle into the same already-loaded real campaign world (the same "tweak real state, then verify the panel" pattern `phase_5_integration_smoke.gd` already established for its own war-score display check) - the battle list, round/hull/sunk detail, and fleet-panel "in battle" line all populate correctly; the retreat button is disabled until the minimum round count, then enabled; pressing it drives the real `RequestFleetRetreatCommand` → `NavalCombatSystem.withdraw_fleet()`, which ends the battle in the defender's favour exactly as combat logic alone would; the final report then shows that exact result.
- **Blockade panel and mission control.** The same test sets a fleet's mission to `blockade` through the dropdown and confirms `SetFleetMissionCommand` actually reaches `WorldState`, and confirms an uncontested home port correctly reads as "not blockaded" through the new label.
- **Naval battle marker.** `conflict_marker_layer_smoke.gd` (extended) confirms an active naval battle produces exactly one additional marker/cluster alongside the pre-existing land battle+siege fixture (they never merge, proving the two marker families are genuinely independent), that clicking it emits a `naval_battle`-typed selection, and that the click correctly opens `NavalHUD` and selects that exact battle. Removing the battle clears its marker batch. The pre-existing land-marker exact-count assertions (`debug_battle_count() == 3`, `debug_marker_instances() == Vector2i(1,1)`, `debug_draw_count() == 2`, `debug_priority_order()`) are untouched by any of this.
- **Stress/performance/global-coast.** `naval_battle_blockade_stress_smoke.gd` (new): 29 real N0.3 fixture ports spanning several real coastlines (not one Channel choke point), each with a co-located hostile pair triggering a real naval battle and a separate hostile fleet blockading from that port's own real sea exit (`MaritimeGraph.port_exits()`, not an assumed shared zone), ring-wired into 10 synthetic wars so every port has a resolvable hostile relationship. Twenty days of concurrent `FleetMovementSystem`/`NavalCombatSystem`/`BlockadeSystem` ticking produced 35 real battles and left 19 of 29 ports blockaded; `BlockadeSystem.all_blockaded_provinces()` - the exact global-coast query N5.3's map overlay/outliner/economy hooks all call - found them in ~30ms. A full save/load round trip afterward passed cleanly, meaning every structural validator this session's naval work touches (`_validate_naval_data`, `_validate_naval_battle_data`, the war-participants check, `_validate_transport_data`) accepted the result of genuinely concurrent battles and blockades at this scale, not just the small hand-built fixtures each mechanic's own correctness test already uses. Timing itself is a smoke-level guard only (90s/20-day budget, generous given this session's sandboxed environment measured 2-3x slower wall-clock time than every pre-existing stress smoke's own budget too, on completely unrelated tests - not something this slice's changes caused).

## Bugs found and fixed while building this (test-driven, not incidental)

- **The fleet mission dropdown was unusable.** `_refresh_mission_validation()` originally re-synced the dropdown's selection to the fleet's *current* mission on every call, including the one wired to the dropdown's own `item_selected` signal - so picking "blockade" would immediately snap back to showing "idle" before the player could ever press "Set mission." `naval_hud_integration_smoke.gd`'s new mission-control assertions caught this on the first run (world state never changed after the simulated click). Fixed by splitting the sync-on-fleet-switch behaviour (`_sync_mission_selection()`, called only from `_refresh_fleet_details()`) from validating whatever is currently selected (`_refresh_mission_validation()`, touches only enablement/tooltip) - the same separation `_populate_admiral_options()`/`_refresh_admiral_validation()` already modeled correctly for the admiral dropdown.
- **`NavalCombatSystem._begin_retreat()`'s "no legal retreat" branch orphaned ships.** Found while building the stress fixture's save-round-trip check, not the battle panel itself: destroying a fleet with no legal retreat destination erased the fleet record but left its still-alive ships in `ship_registry`, referencing a `fleet_id` that no longer existed - `_validate_naval_data`'s ship loop would have rejected that exact save (`"Ship references unknown fleet"`). Fixed by erasing the fleet's ships alongside the fleet record, matching how `_finish_battle()`'s own destruction path (via `_remove_sunk_ships`) already keeps ships and fleets in lockstep.
