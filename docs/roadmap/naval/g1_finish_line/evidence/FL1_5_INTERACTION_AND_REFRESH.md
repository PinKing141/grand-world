# FL1.5 Closure - Interaction and Refresh

**Status:** Complete. This closes FL1's own last open item.
**Satisfies:** FL1.5 ("Interaction and refresh") from [01 - FL1 Naval Map Presentation](../01_FL1_MAP_PRESENTATION.md), specifically the two edge cases [FL1_FLEET_MARKER_FIRST_SLICE.md](FL1_FLEET_MARKER_FIRST_SLICE.md) had already named and deliberately deferred: NavalHUD-driven reverse sync, and the remaining event-driven-refresh gaps.

## What was found before implementation (per the roadmap's own rule 1)

Re-reading FL1.5's five bullets against the actual current code, three were already genuinely done (map-click-to-HUD forward sync for both fleets and battles; deterministic cluster ordering via precomputed sort keys - all built in the earlier FL1.1-FL1.4 packets). Two real, concrete, verifiable gaps remained, both already anticipated by name in the prior packet's own "deliberately not built" notes rather than newly discovered here:

- **NavalHUD-driven reverse sync**: `FleetMarkerLayer` pushes map clicks into `NavalHUD` (`naval_hud.select_fleet(...)`), but nothing pushed the other direction. Selecting a different fleet from the panel's own dropdown, or the panel's existing "previous fleet no longer exists, fall back to the sorted-first fleet" reselection logic (`_refresh_fleet_options()`), left `FleetMarkerLayer._selected_fleet_id` - and therefore the drawn route - pointing at whatever was last clicked on the map, silently disagreeing with what the panel now showed.
- **`fleet_scuttled` had no listener anywhere in the UI layer** - not in `NavalHUD`, not in `FleetMarkerLayer`. This is a genuine bug, not just an untested edge case: `ScuttleFleetCommand` (`scripts/simulation/commands/scuttle_fleet_command.gd`) emits it, and the very button that triggers it (`NavalHUD`'s own Scuttle button, `_scuttle_selected_fleet()`) sits in the same panel that failed to react to it. A scuttled fleet's marker stayed on the map, and the panel kept showing its stale details, until some unrelated event happened to trigger a refresh. `fleet_destroyed` - the near-identical "this fleet no longer exists" event for combat losses - was already connected in both files; `fleet_scuttled` was simply missed.

## What shipped

- **`NavalHUD.fleet_marker_layer`** (new `@export`, wired in `scenes/main.tscn`'s `NavalHUD` node to `../FleetMarkerLayer`) - `_refresh_fleet_details()` now calls `fleet_marker_layer.set_selected_fleet(fleet_id)` at its own entry point, the single place every fleet-selection path in the panel already funnels through (the dropdown's `item_selected` callback, the map-click forward-sync path, and `_refresh_all()`'s own fallback reselection all call `_refresh_fleet_details()` already) - one fix point covers every selection-changing path, not a parallel sync call added to each caller individually.
- **`FleetMarkerLayer._connect_events()`** and **`NavalHUD._connect_events()`** both now also subscribe to `fleet_scuttled`, mirroring the existing `fleet_destroyed` handling exactly (map: mark the marker batch dirty; panel: `_refresh_all()`).

## Verification

`tests/fleet_marker_layer_smoke.gd` (extended):
- A HUD dropdown selection change (`fleet_option.select()` + `_refresh_fleet_details()`, with no map interaction at all) is proven to push the new fleet ID out to `FleetMarkerLayer.selected_fleet()`.
- The panel's own fallback reselection (triggered by removing the currently-selected fleet from the registry and calling `_refresh_all()`) is proven to carry the map's selection along with it - `FleetMarkerLayer.selected_fleet()` and `NavalHUD._selected_fleet_id()` are asserted equal afterward, not just individually plausible.
- `fleet_scuttled` is proven to mark the map batch dirty (checked before any forced refresh, so the assertion tests the listener itself, not just the eventual rebuild) and to actually remove the scuttled fleet's marker once rebuilt; the HUD is proven to stop offering the scuttled fleet only after first proving it was genuinely selected beforehand (otherwise the assertion would pass trivially whether or not the fix existed).

**A real, unrelated regression found while running the full FL1.5 verification sweep, not part of this packet's own scope, but fixed as part of verifying it**: `tests/naval_hud_duplicate_action_safety_test.gd` was failing silently - it printed "passed" despite an internal `push_error`/`quit(1)` and a followup "Out of bounds" script error, because Godot's `SceneTree.quit()` does not abort a still-running `await`-based script function; execution continued past the failure to a later unconditional `quit(0)`, which is what the process actually exited with. The underlying failure was real: the panel's default ship selection (`ship_option`'s alphabetically-first item, `heavy_galleon`) requires `military` technology level 1 - a gate [FL3_2_TECHNOLOGY_GATE.md](FL3_2_TECHNOLOGY_GATE.md) correctly added earlier in this effort - which a fresh 1444 ENG country does not have, so both of the test's rapid construction presses were correctly rejected and `naval_construction_registry` stayed empty, not duplicated. The gate itself was working as designed; the test had never been updated to pick an eligible ship. Fixed by explicitly selecting `war_galley` (technology level 0) before the double-press check, the same "deliberately generous" spirit the test's own treasury/sailors setup already uses for cost eligibility - not by weakening the gate or the test's own duplicate-press assertion.

Full regression re-run after all fixes, checked against the established "grep the debug log for hidden script errors, not just the printed pass line" discipline: `fleet_marker_layer_smoke`, `conflict_marker_layer_smoke`, `campaign_interface_shell_smoke`, `naval_hud_integration_smoke`, `naval_hud_duplicate_action_safety_test` all pass clean with zero hidden `SCRIPT ERROR`/`Parse Error`/`Nonexistent function`/`Invalid call`/`Invalid access` lines in their debug logs.

## Deliberately out of scope for this packet

- **Foreign (non-player) fleet click behaviour**: clicking a hostile or third-party fleet's marker opens the panel but selects nothing in its fleet dropdown, since `NavalHUD`'s fleet list is scoped to the player's own fleets by design (`_country_fleets()`). Already named as an inherited, pre-existing scope limitation in [FL1_FLEET_MARKER_FIRST_SLICE.md](FL1_FLEET_MARKER_FIRST_SLICE.md), not a regression and not part of FL1.5's own bullets (which do not distinguish fleet ownership).
- **Save/load reconciliation beyond what already worked**: `world_reloaded` was already connected in both `FleetMarkerLayer` and `NavalHUD` before this packet, and each already degrades safely when the previously selected fleet no longer exists after a load (route simply stops drawing; panel falls back to the sorted-first remaining fleet) - the reverse-sync fix above now also keeps the two in agreement in that case, closing the one part of "save/load" that was genuinely open. No dedicated save/load round-trip test was added beyond the fallback-reselection case already covered above, since no gap beyond that was found.

## What this means for FL1

FL1.1 and FL1.5 are now fully complete with no remaining named gap. FL1.2, FL1.3, and FL1.4 are unaffected by this packet and remain at their own prior "core complete" status - each still has its own separately-named, deliberately-deferred follow-up (proving live selection survives a mid-selection merge/split; a dedicated peace/merge/load-reconciliation route-cleanup test; blockade attacker identification; coloured coastline segments) that this packet did not attempt, since none of them are FL1.5's own bullets.
