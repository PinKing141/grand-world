# FL2.1 - Fleet Summary Panel

**Status:** Complete, including the selection-survival fallback added in a follow-up pass. Targeted tests pass (see Verification).
**Satisfies:** every FL2.1 finding recorded in [FL2_CLOSURE_AUDIT.md](FL2_CLOSURE_AUDIT.md), including "Selection survival on fleet destruction," closed below after the display work.

## Scope decisions

**Name and owner.** `fleet.display_name` is shown, falling back to the raw `fleet_id` when empty - a fleet that has never been explicitly named still shows something stable and unique rather than a blank field. Owner is resolved through `simulation_controller.country_registry.display_name(owner_country_id)`, the same query `country_label_layer.gd` already uses for country names elsewhere in the UI - not a new resolution path. The audit's recommended packet list only mentioned name, location/home-port, admiral, route, class breakdown, and repair state; it omitted owner and crew even though FL2.1's own roadmap bullets explicitly require both ("present fleet name, owner, admiral..." and "...crew, morale..."). Both are included in this packet rather than leaving FL2.1 partial again immediately after closing it.

**Location and home port.** Both now resolve through the panel's existing `_province_name()` helper - the same fallback (`"province %d"` for an ID with no resolvable name) that every other province reference in this file already uses, including sea zones, which are not present in `ProvinceGraph` and so fall back by construction. This is not a new gap: naval battle zones already displayed the identical fallback before this packet.

**Admiral.** Resolved through a new `_admiral_display_name()` helper that matches - not duplicates conceptually, but intentionally mirrors - the exact "name, falling back to the raw ID" convention `_populate_admiral_options()`'s own dropdown already used before this packet. The summary line and the picker now agree on what to call the same person by construction, since both read `character.get("name", admiral_id)`.

**Route and arrival.** The audit flagged that `next_arrival_day` was never shown - but a closer read of `FleetMovementSystem.advance_day()` found that `next_arrival_day` is only ever the *next waypoint's* arrival day, not the final destination's, for any route with more than one remaining leg. Showing it labelled as a plain "arrival" would have been misleading, not just missing. Two things are now shown instead: the untraversed route itself (`fleet.remaining_path.slice(fleet.path_index)`, resolved to names, joined with `->`), and two distinct days - "next waypoint arrival" (`next_arrival_day` as-is) and "final ETA," computed by a new `FleetSystem.route_completion_day()` query that starts from `next_arrival_day` and sums `MaritimeGraph.leg_cost_days()` for every remaining leg beyond it, using the same `speed_multiplier_bp()` `FleetMovementSystem` itself advances by. Both lines are only shown while the fleet is actually `moving` or `retreating`; a docked fleet has no pending route to show.

**Class breakdown.** `FleetSystem.recompute_aggregate()` now computes `family_counts`, one entry per `ShipDefinitions.ship_families()` (`heavy`, `light`, `galley`, `transport`, in that deterministic order), always present even at zero so no caller needs to guard a missing key. A new `format_class_counts()` renders it as `"2 heavy, 1 transport"`, omitting zero-count families. A second, smaller query, `class_counts_for_ships()`, computes the same shape for an arbitrary ship-ID list rather than a fleet's full membership - used to give the FL2.2 organisation preview (see below) real data without waiting for a dedicated "resulting fleet" packet.

**Crew readiness.** `recompute_aggregate()` now computes `crew_readiness_bp`, a **sailor-cost-weighted** average of every ship's `crew_bp`, not a flat average. A fully-crewed war galley (150 sailors) says more about a fleet's real readiness than a fully-crewed transport cog (60 sailors); an undercrewed galley should move the fleet-wide number more than an equally undercrewed cog would. An empty fleet (zero total sailor cost) defaults to full readiness (10000bp) rather than dividing by zero, matching `hull_pct`'s own "100% if there's nothing to be damaged" convention already established in this file.

**Repair state.** A new `FleetLogisticsSystem.repairing_ship_count()` reads the same authoritative per-ship `repairing` boolean `_repair_one_ship()` and `_apply_attrition()` already set and clear - the panel does not re-derive repair status from hull/crew thresholds itself. Shown as `"Repairing X/N ships"`, and only appended when at least one ship is actually repairing, independent of whether `fleet.mission == "repair"` - the exact gap the closure audit found: a fleet passively healing from attrition without ever having its mission set to `repair` previously showed nothing.

## Selection survival on fleet destruction (closed in a follow-up pass)

`_refresh_fleet_options()` previously only ever called `fleet_option.select()` when the previously selected fleet still existed in the rebuilt list. When it didn't (destroyed by scuttle/combat, merged away), nothing called `select()` at all - which fleet ended up shown was whatever Godot's `OptionButton` happens to default to after `clear()` and repeated `add_item()` calls, not a decision this code made. This is now an explicit branch: if the previous selection is gone and fleets remain, selection moves to `fleet_ids[0]` - `_country_fleets()`'s own sorted order, the same "lowest sorted ID" convention `MergeFleetsCommand`'s survivor pick already established elsewhere in this codebase - so the outcome is deterministic and testable rather than an engine implementation detail. The trivial "zero fleets left" case was already handled correctly before this change and needed no fix.

## FL2.2 organisation preview (small, reused data)

The FL2.2 evidence doc's own open bullet - "preview the resulting class mix, speed, capacity and mission impact" - needs the same class-breakdown data this packet just added, plus work this packet does not do (simulating a fleet that doesn't exist yet). Building the full preview remains a future packet. What ships now: `OrganisationLabel`'s text updates live as ships are selected in the split/transfer list, showing the class mix of exactly what's selected (e.g. `"Selected 2 ships (1 galley, 1 transport) - split into a new fleet or transfer to the target fleet."`), reusing `class_counts_for_ships()`/`format_class_counts()` directly. This is a real, immediately gradable improvement - "what you are about to peel off" - without claiming the full resulting-fleet preview is built.

## Verification

- `tests/naval_fleet_summary_test.gd` (simulation layer): a mixed-family fleet's `family_counts`/`format_class_counts()` output; `class_counts_for_ships()` agreeing with `recompute_aggregate()`'s own family_counts for the same ships; sailor-cost-weighted `crew_readiness_bp` against a hand-computed expected value (distinguishing it from what a flat average would produce); the empty-fleet zero-division default; `repairing_ship_count()` counting only flagged ships while the fleet's mission stays `idle`; and `route_completion_day()` across three snapshots of a real three-leg synthetic route (order time, mid-traversal, final leg) proving the final ETA stays internally consistent regardless of how far the fleet has progressed, plus the docked-fleet `-1` case.
- `tests/naval_fleet_summary_hud_test.gd` (headless `main.tscn` integration): a real `NavalHUD.fleet_details_label` read for a named, mixed-family fleet - resolved name, owner, location, home port, class breakdown, crew percentage; a second, unnamed fleet falling back to its raw `fleet_id`; a real `AssignAdmiralCommand`-equivalent fixture showing the admiral's resolved name, not the raw character ID; passive repair text appearing with the mission left at `idle`; and a real two-leg `MoveFleetCommand` order (Calais -> Straits of Dover -> Kent, matching `naval_fleet_movement_test.gd`'s own fixture) whose route/next-waypoint-arrival/final-ETA text is checked both at order time and after two ticks of real partial traversal.
- Both tests pass headlessly under Godot 4.7.1 and are registered in `tools/testing/run_all_tests.py`.
- Existing tests updated to match the new resolved-name text: `tests/naval_fleet_home_port_hud_test.gd`'s two home-port assertions now check for `"Home port Calais"`/`"Home port Kent"` instead of the raw province IDs the old text showed.
- Targeted naval regression (`naval_fleet_organisation_test`, `naval_fleet_organisation_hud_test`, `naval_fleet_home_port_hud_test`, `naval_fleet_scuttle_test`, `naval_fleet_scuttle_hud_test`, `naval_hud_integration_smoke`, `naval_fleet_logistics_test`, `naval_fleet_state_test`, `naval_admiral_test`, `naval_combat_test`, `naval_ai_test`, `naval_ai_organisation_test`, `naval_channel_release_gate_test`) re-run clean after extending `recompute_aggregate()`'s output shape, confirming no other system depended on the aggregate dictionary's exact prior key set.
- `tests/naval_fleet_selection_fallback_hud_test.gd` (headless `main.tscn` integration): deliberately selects the alphabetically-*last* of three fixture fleets (so an implicit "index 0" engine default would already coincidentally look correct without this fix), scuttles it, and confirms selection explicitly moves to the sorted-first survivor with a real, non-stale panel refresh; repeats once more to prove the fallback cascades correctly on a second destruction; and confirms the trivial zero-fleets-left case still ends in the panel's real empty state. Registered in `tools/testing/run_all_tests.py`.
