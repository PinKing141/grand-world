# FL6 - Automatable Acceptance Slice

**Status:** Validation. The automatable pieces below pass; the rest of FL6's exit gate is inherently manual (hands-on hardware, a real display/GPU, colour-vision-deficient human review, a reviewer's sign-off) and is explicitly not attempted here - flagged, not silently skipped.
**Satisfies:** the parts of [06 - FL6 Rendered and Accessibility Acceptance](../06_FL6_RENDERED_AND_ACCESSIBILITY_ACCEPTANCE.md) that can be proven headlessly, without a human at real hardware.

## Method

FL6's own scope is written as a hardware/human acceptance gate (RenderDoc captures, reviewer name, screenshots, "run on the current low-end Intel UHD 600-class laptop"), not a packet of code to build. Before writing anything, each FL6.1-FL6.4 bullet was checked against what actually exists in the codebase today (`scripts/ui/naval_hud.gd`, `scenes/ui/naval_hud.tscn`, and the wider UI stack), the same "read the real code first" discipline the FL2 closure audit used - not assumed complete, and not assumed automatable just because it would be convenient.

## FL6.1 Resolution and UI scale - partially automated

- **Test 1366x768, 1920x1080, and other required layouts.** Done for two of these two named resolutions plus the two breakpoints `tests/ui_layout_smoke.gd` already covered (1700x960, 1152x648) - `NavalHUD/NavalPanel` was added to that test's containment/overlap checks (it wasn't in there at all before this packet).
- **A real bug found and fixed in the process:** at every one of those four resolutions, `NavalPanel` was rendering **off-canvas** - `[P: (649.0, -212.0), S: (622.0, 1508.0)]` against a `[0,0]-[1920,1084]` canvas at 1700x960, meaning the panel's top edge sat 212px above the visible screen. Root cause: `NavalPanel/Margin/Column` is a plain `VBoxContainer` with no scrolling, and its natural minimum height had grown to ~1508px over the course of this session's FL1/FL2 work (fleet actions, admiral, construction, transport, organisation, mission, battle, and blockade sections all stacked in one column) - far past the panel's nominal 460px anchor-offset box and past the 1080px design canvas itself. Godot's `Container` base class enforces "at least minimum size" even against an anchored offset box, so the panel kept growing and re-centering around its own inflated height instead of respecting its intended box. **Fixed**: `NavalPanel/Margin/Column` is now wrapped in a `ScrollContainer` (`NavalPanel/Margin/Scroll/Column`), matching the exact pattern `CampaignInterfaceShell._ensure_province_scroll()` already established elsewhere in this codebase for the same "a panel's content outgrew its box" problem - horizontal scrolling disabled, vertical scrolling absorbs any future growth instead of pushing the whole panel off-screen. Every `%UniqueName` node reference in `naval_hud.gd` continued to resolve correctly (Godot's `%` lookup is scene-wide, not path-relative), so no script changes were needed.
- **Test every supported UI-scale value.** Not attempted. This project has no bespoke UI-scale setting/slider anywhere in the codebase (grep across `scripts/`/`scenes/` for `ui_scale`/`content_scale_factor` turns up only design-doc references, never an implementation) - there is no feature here to test yet. Godot's own engine-level `content_scale_factor` project setting was not exercised either, since interacting correctly with this project's "stretch canvas" sizing convention (`ui_layout_smoke.gd`'s own `canvas_rect`/`Rect2(Vector2.ZERO, Vector2(root.get_visible_rect().size))` pattern) under a non-default scale factor needs dedicated verification this packet did not have room for - flagged, not silently assumed fine.

## FL6.2 Input and focus - partially automated

- **Complete primary naval workflows using mouse only.** Every `tests/naval_*_hud_test.gd` file already does exactly this (driving real `Button`/`OptionButton`/`ItemList` controls through a real, loaded `main.tscn`), across roughly a dozen test files built over FL1/FL2. Already covered, not newly built here.
- **Complete the same workflows using keyboard only. Define stable focus order and visible focus state. Ensure Escape closes only the top modal and never issues or cancels an order.** **Not built, and not naval-specific.** A full survey found `naval_hud.gd` has zero custom keyboard input handling (`_input`/`_unhandled_input`/`_gui_input`/`InputEventKey` - none exist), and `naval_hud.tscn` has no `focus_neighbor_*`/`focus_mode` overrides on any of its ~20 focusable controls - it relies entirely on Godot's default automatic Tab-order chain. This is true of **every gameplay HUD panel in the game**, not a naval regression: `economy_hud.gd`, `war_hud.gd`, `character_hud.gd`, and `country_depth_hud.gd` all have the identical gap. Escape-key handling exists in exactly three places in the whole codebase - `main_menu.gd`, `country_selection_screen.gd` (menu-screen panel close), and `province_selector.gd` (clears map selection) - none of them a gameplay HUD panel, and there is no modal/dialog system anywhere in the game to "close the top" of (confirmed: zero `PopupPanel`/`AcceptDialog`/`ConfirmationDialog` instances exist in `scenes/ui/`). Building real keyboard-only navigation and an Escape-close convention is a whole-UI input-model decision spanning every panel in the game, not a naval-specific fix this FL6 packet should invent unilaterally. Recommend treating it as its own deliberately-scoped initiative (with an owner decision on the actual convention) rather than a silent, naval-only patch.
- **Verify double-click, rapid-click and key-repeat cannot duplicate commands.** Automated: `tests/naval_hud_duplicate_action_safety_test.gd` calls every naval action handler **twice in a row** with no refresh in between (the real race a rapid double-press or key-repeat creates: two commands landing in the same `SimulationScheduler` batch before `process_commands()` runs and the button's own `disabled` state catches up), then processes the batch once and checks the world ends up exactly as if the button had been pressed once. Covers the two highest-risk cases directly (treasury/sailor double-spend on ship construction - guarded by `ConstructShipCommand`'s existing one-project-per-port queue limit, not just cost; capacity double-reservation on embark - guarded by the army's own status flipping to `embarking` after the first command applies, which the second command's `validate()` then rejects), the three organisation commands' ownership-integrity invariant (split/transfer/merge, "no ship may be lost or duplicated"), and a cancel-then-cancel-again case for transport.

  The remaining action buttons (move, retreat, cancel-movement, set-home-port, set-mission, assign-admiral) were **audited, not individually re-tested**, since each already falls into one of two structurally safe categories, verified by reading `validate()`/`apply()` directly rather than assumed:
  - *Naturally rejected once the first command's `apply()` has already changed the precondition the second command's `validate()` checks*: `RequestFleetRetreatCommand` (`apply()` clears `battle_id`; the second call's `validate()` then rejects with "The fleet is not in a battle"), `CancelFleetMovementCommand` (`apply()` moves `location_status` out of `moving`; the second call's `validate()` then rejects with "The fleet is not moving").
  - *A pure state-overwrite where re-issuing the identical order is a harmless idempotent no-op, not corruption*: `MoveFleetCommand` (a second move to the same destination just recomputes the same route from the fleet's current position), `SetFleetHomePortCommand` and `SetFleetMissionCommand` (re-setting the same target value twice), `AssignAdmiralCommand` (already explicitly proven idempotent by `naval_admiral_test.gd`'s own "re-assigning the same admiral to the same fleet must be accepted" case).

## FL6.3 Colour, text and motion - not automated, one structural note

- `naval_hud.gd` never uses `modulate`/`self_modulate`/`font_color`/raw `Color` values anywhere - every status in the fleet panel (supply, hull, mission, blockade) is conveyed through text (percentages, words like "yes"/"no", capitalized status names), so "pair colour with icons, shapes, labels or percentages" is true by construction for this specific panel, since it never uses colour alone in the first place. This is a structural observation, not a substitute for the real requirement: contrast and colour-vision-profile checking for the *map-side* colour-coded elements (fleet markers, blockade tiers, country colours, all rendered via `ConflictMarkerLayer`/`FleetMarkerLayer` shaders) still needs actual human colour-vision-deficient review on a real display, which this packet cannot perform.
- Reduced-motion behaviour and animation-free battle/blockade comprehension were not assessed - both need a human watching the actual rendered game, not a headless check.

## FL6.4 Hardware and renderer - not automated

- "Run on the current low-end Intel UHD 600-class laptop," "run on the project's target/reference Windows hardware," "exercise the approved Godot renderer paths," and RenderDoc frame captures are all inherently hands-on-hardware work with no headless equivalent - not attempted.
- "Confirm rendered and headless authoritative checksums remain equal": `tests/simulation_frame_rate_determinism_test.gd` already proves the simulation checksum is identical across different `_process()` frame rates (30 FPS vs 120 FPS) for a real naval scenario (fleet movement, a full embark/sail/disembark transport operation), and every naval HUD integration test in this session loads the complete, real `main.tscn` scene graph (3D meshes, shaders, camera, every HUD panel) under `--headless` and still produces correct, checksummed behaviour - strong indirect evidence the simulation is presentation-independent by construction, consistent with this project's own architecture rule ("UI and AI must not mutate registries directly; presentation only interpolates"). This is not the same claim as "a real windowed/GPU-attached run produces the identical checksum to a headless run" - that specific comparison needs an actual windowed process on real hardware to run once, which this environment cannot do.

## Verification

- `tests/ui_layout_smoke.gd` (extended, not new): `NavalHUD/NavalPanel` added to the containment/overlap checks; 1366x768 and 1920x1080 added to the resolution sweep. Passes at all four resolutions after the `ScrollContainer` fix.
- `tests/naval_hud_duplicate_action_safety_test.gd` (new): rapid double-press safety for construct-ship, embark, split, transfer, merge, and cancel-transport. Passes.
- Both registered in `tools/testing/run_all_tests.py`.
- Targeted naval HUD regression (`naval_hud_integration_smoke`, `naval_fleet_organisation_hud_test`, `naval_fleet_home_port_hud_test`, `naval_fleet_scuttle_hud_test`, `naval_fleet_summary_hud_test`, `naval_transport_workflow_hud_test`, `naval_fleet_selection_fallback_hud_test`, `campaign_interface_shell_smoke`) re-run clean after the `naval_hud.tscn` scene restructure, confirming the new `ScrollContainer` didn't disturb any existing `%UniqueName` control reference or button behaviour.

## What FL6 needs next (not done here)

1. A real acceptance walkthrough on actual target/reference hardware, by a human, per the roadmap's own "Acceptance walkthrough" and "Evidence record" sections (device, GPU/driver, renderer, resolution, UI scale, reviewer name).
2. A deliberate, owner-level decision on whether/how to build keyboard-only navigation and an Escape-close convention for the whole game's HUD stack - not a naval-only patch, since the gap is game-wide.
3. Colour-vision-deficient human review of the map's colour-coded elements (fleet/blockade/battle markers, country colours).
4. If a UI-scale feature is ever built, layout verification at each supported scale value, extending `ui_layout_smoke.gd`'s existing pattern.
