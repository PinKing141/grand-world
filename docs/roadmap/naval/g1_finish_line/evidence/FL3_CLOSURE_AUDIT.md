# FL3 Closure Audit and Scope Reconciliation

**Status:** Audit complete. This document corrects the roadmap's own status line: [03_FL3_NAVAL_AI_COMPLETION.md](../03_FL3_NAVAL_AI_COMPLETION.md) says `Validation` ("implementation exists and targeted tests pass, but required rendered, manual, stress, or full-suite evidence is incomplete" per the roadmap's own status definitions), but the actual implementation in `scripts/simulation/naval_ai_system.gd` is substantially narrower than the FL3.1-FL3.6 exit gate describes - not a case of untested-but-complete work, but of real, undone design surface. Unusually, this is not a "previously overstated" finding the way FL2's audit was: `naval_ai_system.gd`'s own header and per-function doc comments are already candid about the narrowing ("Scope is deliberately narrower than 06_N6's full 'AI Planning Layers' section - see each `_plan_*()` function's own doc comment for exactly what it does and does not attempt"). The roadmap status line simply never caught up to what the code itself already admits.
**Method:** `scripts/simulation/naval_ai_system.gd` (632 lines) read in full; every FL3.1-FL3.6 bullet checked against actual function bodies; `tests/naval_ai_test.gd`, `tests/naval_ai_threat_test.gd`, `tests/naval_ai_transport_test.gd`, and `tests/naval_ai_organisation_test.gd` read in full for what's actually proven versus assumed.

## Corrected status

`FL3` should read **Complete**, not `Validation` - all six sub-scopes (FL3.1-FL3.6) and all four "Automated verification" roadmap claims (battle arbitration, recovery matrix, trace-neutrality, performance budget) are now complete, including FL3.5's proactive escort reservation and escort-follows-the-voyage, and FL3.2's naval maintenance command. See each packet's own evidence doc, most recently [FL3_VERIFICATION_4_PERFORMANCE_BUDGET.md](FL3_VERIFICATION_4_PERFORMANCE_BUDGET.md) and [FL3_2_NAVY_MAINTENANCE.md](FL3_2_NAVY_MAINTENANCE.md). FL3 has no remaining named gap.

## Per-bullet breakdown

### FL3.1 Threat and opportunity map - complete
- ~~"Build a country-relative sea-zone assessment from hostile power, friendly support, recent battles, ports, blockades, supply distance and transport stakes."~~ - fixed: `NavalThreatMap.assess()` now computes six of these seven as distinct fields (`hostile_power`, `friendly_power`, `recent_battle_bp`, `has_blockade_target`, `transport_stake`, `supply_days`); "ports" is deliberately folded into `supply_days` rather than kept as a redundant separate boolean. See [FL3_1_THREAT_OPPORTUNITY_MAP.md](FL3_1_THREAT_OPPORTUNITY_MAP.md).
- ~~"Separate authoritative inputs from derived cache data."~~ - fixed: `NavalThreatMap._compute()` (authoritative, no cache awareness) is now cleanly separated from `assess()`'s cache bookkeeping (which never itself decides anything).
- ~~"Define cache revision and invalidation for fleet, war, access, ownership and port changes."~~ - fixed: a day-boundary-plus-explicit-`invalidate()` cache contract, with the reasoning for why day-boundary alone already covers all five named categories documented in the evidence doc. True intra-day event-triggered invalidation remains open, tracked under FL3.4 below (not duplicated here).
- **PRESENT** (confirmed, unchanged): "Use stable ordering and integer scoring." Still a plain deterministic `int`-valued computation throughout.
- **Not a gap - whole-game architectural choice** (confirmed, unchanged): "Keep an interface that can later filter knowledge through naval visibility." No fog-of-war/visibility/detection system exists anywhere in `scripts/simulation/`, for naval or land AI. Not scored as missing.

### FL3.2 Strategic posture and force plan - complete
- ~~"Classify maritime countries as peace, threatened, wartime, invasion, recovery or expansion posture."~~ - fixed: all six now classified. See [FL3_2_STRATEGIC_POSTURE.md](FL3_2_STRATEGIC_POSTURE.md) for the precedence chain and the rivalry-vs-hostile-power distinction discovered while building "threatened."
- ~~"Calculate an affordable heavy, light, galley and transport target mix."~~ - fixed: `POSTURE_SHIP_MIX_BP`, a posture-keyed basis-point mix, now drives which family gets built next.
- ~~"Respect treasury reserve, sailors, maintenance, port capacity, technology and land-war needs."~~ - fixed for treasury (unchanged, now shared), sailors (now checked proactively, not just inherited from command rejection), port capacity (confirmed already true), land-war needs (shares land AI's own reserve formula), technology gating - a genuine authoritative correctness gap, not just an AI-planning one (`ConstructShipCommand.validate()` never checked a ship's `required_technology` field at all, for any caller), fixed by mirroring `RecruitUnitCommand`'s own country-depth pattern exactly (see [FL3_2_TECHNOLOGY_GATE.md](FL3_2_TECHNOLOGY_GATE.md)) - and maintenance. **The maintenance gap's first read here was wrong, and was corrected before building anything, not after**: this audit originally found naval's `maintenance_posture_bp` field (`campaign_world_state.gd:209`, only ever read as a combat-readiness penalty in `naval_combat_system.gd`, never connected to any cost) and concluded the economic half was missing entirely. Checking land's own `army_maintenance_bp` more closely revealed a *second*, separate, country-wide field already existed for navy - `navy_maintenance_bp` - already fully wired into the `navy_maintenance` ledger line (`economy_system.gd:253,322-327`), exactly mirroring `army_maintenance_bp`'s own shape. The economic connection was never missing; only a command to change it and AI logic to use it were. `SetNavyMaintenanceCommand` and `NavalAISystem._consider_navy_maintenance()` now close this in full. See [FL3_2_NAVY_MAINTENANCE.md](FL3_2_NAVY_MAINTENANCE.md). `maintenance_posture_bp` is a genuinely separate, still-unaddressed concept (confirmed by checking that `army_maintenance_bp` has no combat-power connection either) and remains its own distinct, out-of-scope item.
- **PARTIAL** (unchanged, confirmed still accurate): "Avoid construction spam through stable slots and cooldowns." No explicit cooldown field exists; `CONSTRUCTION_INTERVAL` remains the only spacing mechanism. Out of this packet's scope (posture and mix, not queue-spam mechanics).
- **PRESENT** (unchanged): "Record why construction was selected or rejected."

### FL3.3 Fleet organisation, basing and reinforcement - complete
- **PARTIAL** (unchanged): "Group ships into task fleets using normal split, merge and transfer commands." Only `MergeFleetsCommand` is ever used. `SplitFleetCommand` and `TransferShipsCommand` are still never invoked by the AI.
- **PRESENT** (unchanged): "Avoid splitting below mission viability... or reorganising active transports."
- ~~"Choose and change home ports based on access, repair, supply, threat and objective distance."~~ - fixed for the one mechanically real case: `_consider_home_port()` reassigns a fleet once its current home port loses basing rights. Discovered while scoping this, not assumed: `home_port_id` has no other downstream effect in the simulation, so the richer repair/supply/threat/objective-distance model would tune a field nothing else reads. See [FL3_3_FL3_5_REINFORCEMENT_HOMEPORT_DANGER.md](FL3_3_FL3_5_REINFORCEMENT_HOMEPORT_DANGER.md).
- ~~"Keep repair and reserve fleets available."~~ - confirmed already true by construction, not assumed: `EconomySystem._complete_naval_construction()` already deposits every newly-built ship into a per-port reserve fleet, and tracing every tactical `_consider_*` function shows a solo docked reserve fleet is never touched (all require `AT_SEA` except repair/return, which only acts on damage/unsupply). It graduates into service via merge or admiral assignment, never risked prematurely. See [FL3_3_ORGANISATION_AND_RESERVES.md](FL3_3_ORGANISATION_AND_RESERVES.md), which also gives split a real use: separating a mixed fleet's combat ships from its transport ones before a voyage, so they are not needlessly tied down. `TransferShipsCommand` still has no concrete trigger found and remains honestly unused.
- ~~"Compare reinforcement arrival time and value before joining a battle."~~ - fixed: `_consider_reinforcement()` orders an idle fleet toward an active battle its side is currently losing, within a bounded arrival window. No new "join" command was needed - `NavalCombatSystem._join_reinforcements()` already auto-adds any fleet sharing a battle's zone.

### FL3.4 Tactical mission decisions - all eight missions now assigned; power still raw attack, not effective power
- ~~"Score patrol, interception, coast protection, blockade, escort, repair, retreat and idle-return candidates."~~ - fixed: `patrol`, `intercept`, `protect_coast`, and `escort` (mapped to `protect_transport`) now have real assignment logic alongside the pre-existing `blockade`/`repair`/`retreat`/`return_to_port`. A related gap found and fixed in the process: none of the four newly-real missions had any completion condition anywhere in the simulation layer, so a fleet assigned one would have carried it forever - `_consider_mission_completion()` now stands a fleet down to idle once its reason no longer holds. See [FL3_4_TACTICAL_MISSIONS.md](FL3_4_TACTICAL_MISSIONS.md).
- **PARTIAL, and narrower than named**: "Compare effective power rather than ship count." `_side_power()` sums each fleet's `aggregate.total_attack` - a real per-side comparison, better than raw ship count, but *not* the "effective power" the roadmap itself defines two lines later: "Include hull, crew, morale, class matchup, positioning estimate, supply, transport value, support arrival and retreat-port availability." None of those eight modifiers are applied - a fleet at 10% hull and a fleet at 100% hull with identical `total_attack` register as equally powerful to `_consider_retreat()` and `_consider_blockade_or_evade()`.
- **PARTIAL**: "Define conservative safety bounds and explicit desperation exceptions." `RETREAT_POWER_RATIO_BP`/`EVADE_POWER_RATIO_BP` are real, explicit, conservative bounds. "Desperation exceptions" (a cornered fleet with no legal retreat behaving differently than one that has options) do not exist as an AI-layer decision - `RequestFleetRetreatCommand`'s own "no legal retreat -> the fleet is simply destroyed" is a simulation-layer consequence the AI never reasons about in advance.
- ~~"Avoid daily full replanning; use staggered schedules and event-triggered invalidation."~~ - fixed: staggered per-country schedules were already real and correctly implemented (`_due()`/`_next_due_day()`, slot-offset intervals); `NavalAISystem` now also subscribes to `naval_battle_started`/`fleet_moved` and forces an immediate off-schedule tactical reconsideration for any country with a fleet in the touched zone, with a new `naval_ai_event_replans` counter making a replan storm measurable. See [FL3_4_EVENT_TRIGGERED_REPLANNING.md](FL3_4_EVENT_TRIGGERED_REPLANNING.md).

### FL3.5 Autonomous transport objectives - complete
- **PRESENT**: "Select a legal army, origin, destination and land objective." `_plan_transport()` reuses land AI's own `target_province_id` faithfully, confirmed by `tests/naval_ai_transport_test.gd`'s real coverage (unreachable-by-land objective correctly triggers a real embark; a land-reachable one correctly does not).
- ~~"Confirm capacity, escort, route, access, supply and acceptable danger before reserving."~~ - fixed: capacity, route, and access were already confirmed; ~~"acceptable danger"~~ fixed via `_route_too_dangerous()`; ~~escort~~ now fixed too - `_plan_transport()` proactively reserves an idle same-port fleet the instant a transport operation is created, and a new `_consider_escort_follow()` chases the escorted operation's current zone once they part ways instead of sitting still or being abandoned. Supply is not explicitly checked, but this is not a real gap - `CreateTransportOperationCommand.validate()` doesn't require supply itself. See [FL3_3_FL3_5_REINFORCEMENT_HOMEPORT_DANGER.md](FL3_3_FL3_5_REINFORCEMENT_HOMEPORT_DANGER.md) and [FL3_5_ESCORT_LIFECYCLE.md](FL3_5_ESCORT_LIFECYCLE.md).
- **PRESENT**: "Create the transport through the shared operation command." Real `CreateTransportOperationCommand` usage, no parallel path.
- **PRESENT, by inheritance not by design**: "Monitor interception, capacity loss, destination capture, access loss and peace." A legitimate claim, but worth stating precisely: this is not naval-AI-specific monitoring, it is `TransportSystem`'s own N3.3 failure-recovery machinery (already part of this project's "trusted" list per the roadmap README), which applies identically to every transport operation regardless of who created it. The AI gets this for free by using the shared command, exactly as intended - correctly claimed, not overclaimed.
- **PRESENT, by inheritance not by design**: "Hand a successful disembarkation back to land AI exactly once." Also correct as claimed: `TransportSystem` returns the army to `ARMY_STATUS_IDLE` on completion, and `StrategicAISystem._issue_army_orders()` already scans every idle army generically on its own schedule. No double-pickup risk (an idle army is picked up once, by whichever system's turn comes next) and no explicit handoff code is needed. Not independently tested by a naval-side test, but the mechanism is a natural consequence of two already-tested systems (`naval_ai_transport_test.gd` for embark, `phase_6_ai_test.gd` for idle-army pickup) rather than new coupled logic.
- **PARTIAL**: "Record missing capacity, escort, route or risk when no plan is issued." Capacity and route rejections are recorded (`"no_transport_capacity"`, `"no_legal_landing"`). Escort and risk are not recorded, for the same reason they are not evaluated - there is nothing to record yet.

### FL3.6 Explainability and bounds - history/counts exist; several named fields and counters do not
- **PARTIAL**: "Record country, day, posture, action, targets, selected score, major rejected candidates, constraints and next planning day" per decision. `_record_decision()` actually stores `day`, `category`, `action`, `score`, `reason`, `alternatives` (`naval_ai_system.gd:577-597`). `country` is implicit (the record lives inside that country's own runtime state, never restated in the record itself - defensible). `posture` is tracked separately on `state["posture"]`, not copied into each individual decision record. `targets` (which fleet, province, or zone a decision concerned) is not a structured field at all - it exists only as free text inside `reason`, not queryable/filterable. `constraints` (what bound, ratio, or threshold actually gated the decision) is not a structured field either. `next planning day` is computed correctly by `_next_due_day()` and surfaced through `debug_snapshot()`, but is not stored inside the decision record itself.
- **PRESENT**: "Bound trace history and candidate counts." `MAX_DECISION_HISTORY := 16` (decision history), rejected candidates capped at 8 (`_record_rejected_candidate()`). Both real, both enforced with `pop_front()`.
- **ABSENT / untested**: "Confirm trace production does not change authoritative results." No test in `tests/naval_ai_*.gd` compares a run with tracing against a run with tracing disabled (there is in fact no way to disable it - recording is unconditional, not gated behind a flag). Given recording only ever writes to `country_runtime[tag]["naval_ai"]`, a country-scoped side channel never read by any *other* system's `validate()`/`apply()`, this is very likely true by construction - but "very likely" is not the same as verified, and there is no flag to even express "tracing off" if it ever needed testing.
- **PARTIAL**: "Add counters for countries planned, candidates evaluated, cache rebuilds and elapsed time." Exactly two global counters exist: `naval_ai_commands_submitted` and `naval_ai_decisions` (both incremented on every successful/every recorded decision respectively). "Countries planned" (how many countries were actually visited this tick, vs. skipped for not being maritime/not due) does not exist. "Candidates evaluated" (as distinct from *decisions*, i.e. rejected candidates counted alongside accepted ones) does not exist as a counter, only as the capped `rejected_candidates` list itself. "Cache rebuilds" cannot exist without a cache (see FL3.1). "Elapsed time" / any per-tick performance counter does not exist at all - no timing instrumentation anywhere in this file.

## What's tested versus assumed

`tests/naval_ai_test.gd` is the only one of the four naval-AI test files that verifies determinism, and it does so genuinely: two independently constructed simulations run 215 real days from the same seed (14441111) against the real 29-port Iberian fixture, then compared for identical ship counts *and* `world.checksum() == world_b.checksum()` - a real two-instance replay, not a hardcoded expectation. `naval_ai_threat_test.gd`, `naval_ai_transport_test.gd`, and `naval_ai_organisation_test.gd` are isolated, single-function unit-style fixtures (hand-built two-to-four-province worlds calling one `_plan_*`/`_consider_*` function directly) - real and valuable for what they cover, but none of them re-verify determinism, and none of the four files together prove any of the roadmap's own four named "Automated verification" claims beyond what's listed above:
- ~~"AI and player targeting the same sea zone still produces one authoritative battle"~~ - fixed: proven through the real command/scheduler/`FleetMovementSystem`/`NavalCombatSystem` pipeline. See [FL3_VERIFICATION_1_BATTLE_ARBITRATION.md](FL3_VERIFICATION_1_BATTLE_ARBITRATION.md), which also records a real, separate, deliberately-deferred gap found while verifying this (a zone with fleets from two different simultaneous wars only ever forms a battle for one of them).
- ~~"AI recovers from destroyed fleets, blocked/captured ports, access loss, peace, debt and insufficient sailors"~~ (the full recovery matrix) - fixed: a genuine before/during/after recovery story proven for five of the six named scenarios (access loss/captured home port was already covered by FL3.3's own test, not duplicated). Two real gaps were found and fixed while building it: `_begin_retreat()`'s no-legal-retreat path never cleared a destroyed fleet's admiral back-reference (closing the gap FL2_5_SCUTTLE_COMMAND.md had already found and left open), and a blockading fleet was invisible to every other tactical decision forever after peace, not merely ineffective. See [FL3_VERIFICATION_2_RECOVERY_MATRIX.md](FL3_VERIFICATION_2_RECOVERY_MATRIX.md).
- ~~"Trace production does not change authoritative results"~~ - fixed: a `naval_ai_tracing_enabled` toggle was added specifically to make this checkable (it did not exist before, so the claim was previously untestable, not just untested), then proven over the real 215-day two-instance Iberian fixture. See [FL3_VERIFICATION_3_TRACE_NEUTRALITY.md](FL3_VERIFICATION_3_TRACE_NEUTRALITY.md).
- ~~"Global planning work is bounded and meets its measured budget"~~ - fixed: `tests/naval_ai_performance_smoke.gd` synthesizes a 20-country roster via `AIDefinitions.from_data()` and measures real elapsed AI-planning time across 65 days against an explicit `AI_PLANNING_BUDGET_MS := 30000.0` budget (measured ~5.75-6.85s), run deliberately last, after split/transfer, event-triggered replanning, and escort lifecycle had all landed and stopped changing planning cost. See [FL3_VERIFICATION_4_PERFORMANCE_BUDGET.md](FL3_VERIFICATION_4_PERFORMANCE_BUDGET.md).

## What this means for FL3's status

FL3 is genuinely `Complete`, not `Validation`. All six sub-scopes (FL3.1-FL3.6) are complete and tested, all four "Automated verification" claims (same-zone battle arbitration, the full recovery matrix, trace-neutrality, the performance budget) are proven, and the last named FL3.2 bullet (naval maintenance) is closed. FL3 has no remaining open item of its own.
