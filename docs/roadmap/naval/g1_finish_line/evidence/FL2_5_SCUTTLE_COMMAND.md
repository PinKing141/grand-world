# FL2.5 - Scuttle Command

**Status:** Complete. Implementation matches the safety-rule design below; targeted tests pass (see Verification).
**Satisfies:** the scuttle half of FL2.5 ("Admiral and scuttle controls") from [02 - FL2 Complete Fleet Management](../02_FL2_FLEET_MANAGEMENT.md).

## Safety rules (defined before implementation)

1. Only the owning player may scuttle.
2. Require the fleet to be docked.
3. Reject fleets moving, fighting, intercepting, or retreating.
4. Reject fleets carrying armies or holding transport reservations.
5. Clean up admiral and mission references.
6. Remove ships deterministically with no refund.
7. Emit an event clearly identifying the loss as scuttling.
8. Require destructive-action confirmation in the UI.
9. Test save/load, duplicate commands, the final national fleet, and deterministic checksums.

## How each rule maps to existing code, and what's genuinely new

**Rules 1-4 (ownership, docked, not moving/fighting/intercepting/retreating, no armies/transport reservations)**: `FleetSystem.is_docked_and_organisable(world, fleet_id, country_tag)` (`fleet_system.gd:74-88`) already checks owner match, not movement-locked, empty `transport_operation_ids`, and `location_status == DOCKED` in one call - the exact same gate `SplitFleetCommand`/`MergeFleetsCommand`/`TransferShipsCommand` already validate through. `DOCKED` excludes `MOVING`, `RETREATING`, `AT_SEA`, and `BATTLE` (fighting) by construction, since those are mutually exclusive location statuses. This covers "moving, fighting, retreating" and "carrying armies or holding transport reservations" (a fleet only carries armies via an active transport reservation, so one check covers both). "Intercepting" is the one condition not covered by `is_docked_and_organisable()` - it is a *mission tag*, not a location status, and a fleet can technically be `DOCKED` while still tagged `mission == "intercept"` (per the FL2 closure audit, `intercept` has no real movement behaviour today, but the mission tag itself persists until changed). `ScuttleFleetCommand.validate()` adds one explicit extra check: reject if `mission == "intercept"`, honouring the rule literally rather than relying on intercept being inert today.

**Rule 5 (clean up admiral and mission references)**: mission cleanup needs no separate step - `mission`, `mission_target_ids`, and `mission_started_day` are fields on the fleet record itself, and the fleet record is being erased entirely, so there is no external mission registry to reconcile (`BlockadeSystem` reads fleet state live, it does not maintain a separate "this fleet is blockading X" table). Admiral cleanup is the one place this command does something **no existing fleet-destruction path in the codebase currently does**: `NavalCombatSystem._begin_retreat()`'s "no legal retreat" branch (`naval_combat_system.gd:772-777`) erases a fleet's ships and the fleet record itself, but never clears the erased fleet's admiral's `character.admiral_fleet_id` back-reference - a real, pre-existing dangling-reference gap in combat-driven fleet destruction, discovered while designing this command. `ScuttleFleetCommand.apply()` clears the admiral's `admiral_fleet_id` explicitly before erasing the fleet, matching the reverse-cleanup `CharacterSystem`'s own death-handling code already does when an *admiral* dies (`character_system.gd:296-301`, the mirror-image case). **This pre-existing gap in combat-driven fleet destruction is out of scope for this packet** (fixing it would touch `NavalCombatSystem`, not FL2's UI/command surface) but is recorded here since it was found while building the equivalent cleanup for scuttle, matching this project's practice of flagging discovered issues even when not fixing them immediately.

**Rule 6 (remove ships deterministically, no refund)**: ships are erased from `ship_registry` in sorted order (matching every other bulk-ship operation in this codebase, e.g. `FleetSystem.move_ships()`'s own `sorted_ship_ids`). "No refund" is not a special code path to add - it is the *absence* of the treasury/manpower credit `DisbandArmyCommand` (the land equivalent) explicitly gives (`disband_army_command.gd:36-37`, returns 25% of strength as manpower). Checked against how the simulation already treats combat ship losses: neither `naval_combat_system.gd` nor `fleet_logistics_system.gd` credits treasury or sailors back to a country when a ship sinks in battle - scuttling deliberately matches that existing, established "losses are losses" precedent rather than being uniquely punitive or uniquely generous.

**Rule 7 (a clearly-identified event)**: a new `fleet_scuttled(fleet_id: String, country_tag: String, ship_count: int)` signal, not a reused `fleet_destroyed(fleet_id, battle_id)`. The existing signal-naming convention already distinguishes voluntary player action from combat loss on the land side (`army_disbanded(army_id)` vs `army_destroyed(army_id, battle_id)`) - `fleet_scuttled` follows that exact precedent rather than overloading `fleet_destroyed`'s `battle_id` parameter with a reason string (which the existing "no_legal_retreat" call site already does slightly awkwardly, passing a reason into a battle-ID-shaped parameter - not a pattern worth extending).

**Rule 8 (destructive-action confirmation in the UI)**: this panel has no existing modal/confirmation-dialog infrastructure (`draggable_panel.gd` is the only shared script `NavalPanel` uses, and it's a drag handler, not a modal). A lightweight confirmation will be built for this command specifically: pressing "Scuttle" the first time arms a distinct confirmation state (button text changes to name the fleet and ship count, e.g. `"Scuttle 3 ships? Confirm"`) rather than acting immediately; pressing again while armed submits the command; changing fleet selection, or any world event that invalidates the armed fleet, disarms it. This mirrors the "identify affected ships/armies" half of the rule directly in the button's own confirming state rather than a separate popup, consistent with this panel's existing "no modals" pattern (see FL2.2's evidence doc on why modals were judged unnecessary here) while still requiring a deliberate second action before anything destructive happens.

**Rule 9 (test coverage)**: see Verification below.

## Implementation

- `scripts/simulation/simulation_event_bus.gd`: added `signal fleet_scuttled(fleet_id: String, country_tag: String, ship_count: int)`.
- `scripts/simulation/commands/scuttle_fleet_command.gd`: `ScuttleFleetCommand`, matching the rules above exactly - `validate()` reuses `FleetSystem.is_docked_and_organisable()` plus an explicit `mission == "intercept"` check; `apply()` clears the admiral's `admiral_fleet_id`, erases every ship and the fleet record (no refund), and emits `fleet_scuttled`.
- `scripts/simulation/simulation_controller.gd`: `scuttle_fleet(country_tag, fleet_id)` wrapper submits the command through the normal scheduler path.
- `scenes/ui/naval_hud.tscn` / `scripts/ui/naval_hud.gd`: a `ScuttleFleetButton` in `FleetActionRow` implements the armed-confirmation UI described under rule 8 above. `_refresh_scuttle_validation()` re-validates on every panel refresh (disabling the button and clearing tooltip text the instant the fleet becomes illegal to scuttle, which also protects an already-armed-but-now-illegal fleet, since a disabled button cannot be pressed). `_scuttle_armed_fleet_id` tracks which fleet is armed; `_refresh_fleet_details()` clears it the moment the selected fleet differs from the armed one, so switching fleets (or losing the armed fleet to any other refresh-triggering event) always requires re-arming rather than leaving a stale confirmation live.

## Verification

- `tests/naval_fleet_scuttle_test.gd` (simulation layer, no rendering): validate-rejection matrix (unknown fleet, non-owner, each non-docked `location_status` - moving/battle/retreating/at_sea, intercepting while docked, an active transport reservation); the happy-path `apply()` - fleet and ship records erased, admiral's `admiral_fleet_id` cleared, no treasury refund, `fleet_scuttled` fires exactly once with the correct fleet/country/ship-count, England's national ship and fleet lists shrink by exactly the scuttled fleet's contents; duplicate-command safety - submitting two `ScuttleFleetCommand`s for the same fleet in one scheduler batch applies the first and rejects the second (`command_rejected`), rather than double-erasing or silently no-op'ing; save/load - a scuttle-affected world's checksum is unchanged across a `to_save_dict()`/`apply_save_dict()` round trip, the scuttled fleet does not reappear, and the admiral's cleared assignment survives reload.
- `tests/naval_fleet_scuttle_hud_test.gd` (headless `main.tscn` integration): a real `NavalHUD.scuttle_fleet_button` press sequence - initial unarmed/enabled state, first press arms without submitting anything (fleet still present), the armed button names the ship count, switching the selected fleet disarms it, re-arming and a second press on the same fleet actually submits `ScuttleFleetCommand` through the real scheduler and removes the fleet and its ships, and the panel returns to a clean unarmed state for the surviving fleet afterward.
- Both tests pass headlessly under Godot 4.7.1 and are registered in `tools/testing/run_all_tests.py` (`FL2.5 scuttle command validation, cleanup, duplicates, save/load` and `FL2.5 scuttle armed-confirmation HUD integration`).
- Targeted naval regression (`naval_fleet_organisation_test`, `naval_fleet_organisation_hud_test`, `naval_fleet_home_port_hud_test`, `naval_hud_integration_smoke`) re-run clean after adding the new button, confirming no regression to the existing FL2.2/FL2.3 controls sharing the same panel and refresh path.

## Deliberately out of scope for this packet

- Fixing the pre-existing admiral-cleanup gap in `NavalCombatSystem._begin_retreat()`'s combat-destruction path (discovered above) - that is a combat-system correctness issue, not part of building scuttle.
- A general-purpose modal/confirmation-dialog component for the whole panel - only the scuttle button gets the armed-confirmation treatment described above; other destructive-adjacent actions (cancel movement, cancel transport, retreat) are left exactly as they are today, since this packet's job is scuttle specifically, not a panel-wide UX redesign.
