# FL1.1/FL1.2/FL1.3/FL1.4 (partial) - Fleet Markers, Routes, and Persistent Blockade Cues

**Status:** Three packets complete and verified. Packet 1 covers FL1.1 (marker view model), the core of FL1.2 (fleet markers with clustering), and the fleet-click half of FL1.5 (interaction/refresh). Packet 2 covers the core of FL1.3 (selected-fleet route drawing and mission-target feedback). Packet 3 covers the core of FL1.4 (a persistent, always-on blockaded-coast marker, distinct from the existing manual "show blockade map" overlay). The remaining FL1.5 lifecycle-refresh edge cases (save/load reconciliation, NavalHUD-driven reverse sync) are still open.
**Satisfies:** part of FL1 ("Fleet, route, battle and blockade map presentation") from [01 - FL1 Naval Map Presentation](../01_FL1_MAP_PRESENTATION.md).

## What was found before implementation (per the roadmap's own rule 1)

A read-only survey confirmed **no fleet-marker map presentation existed at all** before this packet. `ConflictMarkerLayer` already renders naval-battle markers (built during the earlier N-pillar work), but nothing rendered individual fleets themselves. `NavalHUD.select_fleet()` existed as dead code with no caller - the fleet panel was reachable only through its own dropdown, never by clicking the map. Two existing systems were close enough templates to reuse directly rather than invent a third pattern:

- `ConflictMarkerLayer._cluster_records()` (already optimised in FL8.3 - precomputed sort keys, native Array sort) - the exact clustering shape FL1.2 asks for ("cluster co-located fleets... open a stable list when a cluster is selected").
- `cartographic_marker_icon.gdshader` - already supports per-instance vertex-colour tinting and renders a cluster-count digit badge automatically from `INSTANCE_CUSTOM.g`, with zero shader changes needed.

## What was built

`scripts/ui/fleet_marker_layer.gd` (new) - `FleetMarkerLayer extends Node3D`, one `MultiMeshInstance3D` batch using the shared "navy" icon (index 1, the same atlas `ConflictMarkerLayer`'s naval-battle markers already use), tinted per-instance by the owner country's registry colour (`CountryRegistry.country_colour()`) rather than a fixed type colour - the same icon, different colour, is what distinguishes a peaceful fleet marker from an active-battle marker at the same location.

- **FL1.1 marker view model**: `_rebuild()` derives one record per fleet directly from `fleet_registry` (`owner_country_id`, `location_id`, `mission`, `location_status`, `battle_id`, `ship_count`/`total_hull`/`total_maximum_hull` from `aggregate`) - a pure read, no new persisted state.
- **FL1.2 clustering**: `_cluster_records()` is a near-verbatim reuse of `ConflictMarkerLayer`'s proven bucketed spatial-clustering algorithm (fold exact co-location first, then cluster nearby screen positions within `cluster_radius_pixels`), adapted to fleet records. Deterministic ordering comes from the same precomputed-key + native-sort shape FL8.3 already established.
- **FL1.5 (fleet click half)**: `marker_at_screen_position()`/`_on_map_click_requested()` reuse the exact click-to-cycle pattern `ConflictMarkerLayer` already uses for battle clusters (repeated clicks on the same cluster advance through its members in signature order) and finally give `NavalHUD.select_fleet()` its caller.
- **Event-driven refresh**: connects to every fleet lifecycle event on the bus (`fleet_created`, `fleets_merged`, `fleet_ships_transferred`, `fleet_home_port_changed`, `fleet_movement_*`, `fleet_mission_changed`, `fleet_destroyed`, `fleet_retreat_started`), the naval battle start/end events (a fleet's `battle_id`/`location_status` changes without a fleet-specific event), and country lifecycle events (`country_formed`, `country_released`, `country_extinct`, `world_reloaded`).
- Wired into `scenes/main.tscn` as a sibling of `ArmyLayer`/`ConflictMarkerLayer`, taking the same `simulation_controller`/`map_render`/`camera_controller` exports plus `naval_hud` (mirroring how `ConflictMarkerLayer` reaches `war_hud`/`naval_hud` for click-through).

## Deliberately not built in packet 1

- **Owner shield-atlas icons** (`ArmyLayer`'s per-country flag rendering): a flat national-colour tint satisfies FL1.2's minimum bar ("icon, outline or text in addition to colour for owner and hostility" - icon is fixed "navy", colour is the distinguishing cue here) without pulling in the full shield-atlas material pipeline. Can be added later without changing the clustering/selection contract.
- **A full cluster member-list popup**: the existing precedent (`WarHUD.focus_conflict_marker()`'s "marker N of M in cluster" text) is click-to-cycle, not a dropdown list. `FleetMarkerLayer` follows the same established pattern rather than inventing a second cluster-browsing UX in the same codebase.
- **Foreign fleet click behaviour**: `naval_hud.select_fleet()` only finds a match in the player's own `fleet_option` list (matching the existing, pre-existing scope of that function - the same scope `select_battle()` already has for naval battles). Clicking a foreign fleet currently opens the panel without selecting anything useful. This is an inherited limitation of the panel, not a regression introduced here, and is a reasonable candidate for its own follow-up packet.

## Packet 2: FL1.3 route and mission-target feedback

`FleetMarkerLayer` gained a selected-fleet concept (`set_selected_fleet()`/`selected_fleet()`, driven today by map clicks - `_on_map_click_requested()` now calls it alongside `naval_hud.select_fleet()`) and derived route geometry for whichever fleet is selected, built by porting `ArmyLayer`'s exact proven dashed-line `ImmediateMesh` approach (`_append_route_segment`/`_append_route_segment_local`/`_append_route_quad`/`_pixels_to_world`, including the same anti-meridian wraparound splitting) rather than inventing a second line-drawing implementation. Cooler blue/teal route colours (vs. `ArmyLayer`'s amber) keep land and sea routes visually distinct when both are on screen together.

- **Route source of truth**: `remaining_path`/`path_index` on the fleet record - the same field `NavalCombatSystem._begin_retreat()` already populates for a forced retreat, so no separate "retreat path" concept was needed.
- **Style distinction** ("Distinguish planned, moving, blocked, retreating and transport routes"): `moving` (plain dashed line), `retreat` (tighter dash, red-orange), `transport` (longer dash, green - detected from a non-empty `transport_operation_ids`, since the carrying fleet's own movement already *is* the transport route once a transport operation starts sailing). `planned`/`blocked` (a player-issued move preview, and an invalid-destination indicator) are not built in this packet - there is no map-click "order a fleet to move" flow yet for a preview to attach to; `set_selected_fleet()` was named and shaped so that flow can add `set_preview_path()`/`set_invalid_destination()` later without changing this packet's contract, the same forward-compatible shape `ArmyLayer` already proves for the land equivalent.
- **Mission target feedback** ("Show mission target zones or ports where a mission has targets"): a ring marker at a blockading fleet's first `mission_target_ids` entry, shown independently of whether the fleet has an active route - "moving toward" and "on station at" are different feedback needs, and an on-station blockader correctly shows the ring with no route at all.
- **Stale-geometry safety**: destroying the currently-selected fleet (sunk in battle, scuttled, etc.) without an explicit deselect correctly clears both the route and the mission-target ring - `_refresh_route()` treats a `_selected_fleet_id` that no longer resolves in `fleet_registry` as "nothing to draw" rather than holding the last-known geometry.

### Deliberately not built in packet 2

- **NavalHUD-dropdown-driven reverse sync**: selecting a fleet via the panel's own dropdown does not yet call `fleet_marker_layer.set_selected_fleet()` (only map clicks do today). `ArmyLayer`'s equivalent (`set_selected_army`) is called from multiple non-map UI sites; the naval equivalent needs a `fleet_marker_layer` export threaded into `NavalHUD` to do the same, deferred to its own follow-up rather than expanding this packet's scope.
- **Planned/blocked (preview) route styles** - see above; the API shape exists, the player-facing move-order flow that would drive it does not yet.

## Packet 3: FL1.4 persistent blockade coast cue

Unlike fleet routes (which belong on `FleetMarkerLayer`, an owned-mobile-entity layer), a blockade is an *event/state* marker in the same category as battles and sieges - `ConflictMarkerLayer` already has three such families (`battle`, `siege`, `naval_battle`), so a fourth (`blockade`) reuses 100% of the existing anchor/cluster/fade/click machinery rather than adding a second marker system for the naval pillar.

- **Source of truth**: `BlockadeSystem.all_blockaded_provinces(world)` for which provinces to mark, `province_blockade_bp()`/`blockade_tier()` for strength - the exact same authoritative query the existing manual "show blockade map" overlay and the economy/repair/construction/siege consumers already use, so the persistent marker and the manual deep-dive overlay can never disagree about which coasts are blockaded.
- **Icon and colour**: the shared marker atlas's "port" icon (index 6 - `ICON_NAMES` in `tools/marker_art/build_marker_assets.py`), tinted with the exact same yellow-to-red bp-intensity formula `NavalHUD._show_blockade_map()` already uses for its own overlay legend, so the two presentations read as the same scale rather than two different colour languages for the same number.
- **Click-through**: `naval_hud.select_blockaded_province(province_id)` (new) reuses the real province-selection path (`_on_province_selected()` -> `_refresh_port_panel()` -> `_refresh_blockade_label()`) that a genuine province click already drives, rather than adding a second "read blockade state" code path.
- **Automatic, not manual**: this marker requires no button press - it is derived and rebuilt on the same dirty-flag cycle as every other `ConflictMarkerLayer` family, refreshed on `blockade_started`/`blockade_ended`/`port_fully_blockaded`/`port_unblocked`/`blockade_level_changed` (all five events 05_N5's own "Events and Queries" work already established).

### Deliberately not built in packet 3

- **Attacker identification**: FL1.4 asks to "show blockade strength/tier and attacker when known." No code anywhere in the naval pillar currently computes "which country is blockading this specific province" as a standalone fact (`blockade_bp_by_side()` needs a war's side lists supplied by the caller, not a reverse lookup) - the existing manual overlay/label don't show it either. This is a real, scoped gap, not an oversight: `BlockadeSystem.blockaded_provinces_for_fleet(fleet_id)` gives the reverse lookup this would need (iterate blockade-mission fleets, check membership), left for its own follow-up rather than bundling a new query concept into a presentation packet.
- **Coloured coastline segments**: FL1.4 says "affected coasts," which could mean colouring the coastline itself, not just a port marker. The marker-only approach was chosen to reuse existing infrastructure exactly; a coastline-highlight pass (if still wanted) is a separate, larger rendering task.

## Verification

`tests/fleet_marker_layer_smoke.gd`: three fixture fleets (two co-located ENG/FRA fleets at a port, one lone ENG fleet at a separate sea zone) prove - marker count matches authoritative fleet count; two co-located fleets collapse into one cluster while the lone fleet stays separate; clicking the port cluster twice cycles deterministically through both members and opens the naval panel with the correct fleet selected; ENG and FRA fleets use genuinely distinct national colours; markers cull at world zoom; clearing every fleet clears every marker. Extended for packet 2: a selected fleet's route style is `moving`/`retreat`/`transport` in the correct circumstances with real line geometry and a destination marker; an on-station blockading fleet shows its mission-target ring with no route and no stale destination marker; destroying the selected fleet or explicitly clearing selection both correctly clear all route/target geometry. Registered in `run_all_tests.py`.

`tests/conflict_marker_layer_smoke.gd` extended for packet 3: a real eligible blockading fleet (a genuine `war_galley` ship, an active war, real province ownership) produces exactly one blockade marker at the blockaded land province - not the fleet's own sea zone - proving the anchor is correct; the marker clusters independently of the (already-cleared) naval-battle marker at the same sea zone; clicking it opens the naval panel with the correct province selected via the real province-selection path; removing the blockading fleet clears the marker.

Full regression suite after all three packets: 70/71 (the sole remaining failure is the already-documented, unrelated FL8.2 hardware-blocked case - see [FL8_2_LABEL_BATCH_BUDGET.md](FL8_2_LABEL_BATCH_BUDGET.md)). The pre-existing `naval_hud_integration_smoke.gd`, `conflict_marker_layer_smoke.gd`, `conflict_marker_stress_smoke.gd` (one apparent 66.69ms-vs-66.67ms budget miss reproduced as noise - two immediate re-runs measured 39ms and 23ms, and the fixture in question creates zero fleets, so the new blockade-collection code path runs its cheapest possible case there), and the full-scale G1 Channel/destructive-lifecycle release-gate tests all still pass, confirming the new marker family and the scene/panel wiring introduced no regression anywhere else.
