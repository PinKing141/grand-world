# FL2.6 - Transport Workflow Integration

**Status:** Complete except "danger," deliberately deferred (see below). Targeted tests pass (see Verification).
**Satisfies:** the FL2.6 findings recorded in [FL2_CLOSURE_AUDIT.md](FL2_CLOSURE_AUDIT.md).

## Scope decisions

**Required, usable, reserved, missing capacity.** Usable and reserved were already fixed in the transport-capacity correctness packet. This packet adds the two still-missing halves: each active operation's own `reserved_capacity` is now shown directly on its line (`"... · N capacity reserved"`), and a **persistent** required/available preview appears whenever an army is selected in the embark dropdown - `"Selected army requires N capacity · M available"` - regardless of whether the current selection would pass or fail validation. Previously these numbers only ever appeared inside `embark_button`'s tooltip, and only once validation had already failed *specifically* for a capacity reason; a legal selection, or a selection blocked for an unrelated reason (no route, wrong province), showed neither number at all.

**Destination and route.** The operation line resolved `destination_province_id` through `_province_name()` instead of printing the raw ID, matching every other province reference in this file. Route required no new simulation work: `TransportSystem`'s existing `apply_transport_movement()`-equivalent step already populates `operation.planned_path` with the real route the moment an operation transitions from embarking to sailing (`transport_system.gd`, the embark->sailing transition) - it was already fully authoritative, just never read by the UI. A sailing operation's line now shows a second "Route ..." line with every waypoint resolved to a name.

**Danger - deliberately not built.** The only existing per-zone risk concept in this codebase is `NavalAISystem._zone_threat()`, a private, AI-internal scoring heuristic never designed for player-facing display or for aggregating risk across a multi-leg route. Reusing it would mean inventing new simulation surface area under a UI packet, not fixing a UI gap - the same reasoning FL2.4's evidence doc already used to defer the inert-mission target pickers to FL3/FL5. Left open, recorded here rather than silently dropped.

**Cancellation.** Two things changed. First, a real correctness bug found while building the consequence tooltip: `CancelTransportOperationCommand.validate()` has always accepted both `embarking` and `disembarking` states, but `_refresh_transport_panel()` only ever set `cancellable_id` for `embarking` - a disembarking operation was silently never offered the button, even though cancelling it has always been legal. Both states are now covered. Second, `cancel_transport_button.tooltip_text` now explains the real, state-specific consequence (`CancelTransportOperationCommand.apply()` read directly): cancelling while embarking returns the army to its origin, unembarked; cancelling while disembarking lands the army immediately at its current location.

**Cross-navigation ("link the fleet, army and operation views").** Scoped to the single-operation case. `transport_label` is one multi-line `Label` rendering every active operation as plain text, not a per-row list with per-row actions - building that would be a larger panel restructure than this bullet needs. A new `FocusCarriedArmyButton` selects the carried army on the map via a new `@export var army_layer: ArmyLayer` (wired in `main.tscn` exactly like `SimulationHUD`'s own existing `army_layer` export - not a new pattern), enabled only when the selected fleet has exactly one active transport operation; with zero or multiple operations it is disabled with an explanatory tooltip rather than guessing which army "focus" should mean. This does not pan the camera (unlike `CampaignInterfaceShell._focus_army()`, which this panel has no `camera_controller` reference to call) - selecting/highlighting the army on `ArmyLayer` is the "link," matching that layer's own existing selection-highlight behaviour.

**Battle-pause, recovery, and peace UI preservation.** The audit's own wording called this "an unverified assumption, not true by construction," since `_connect_events()` had no `war_declared`/`peace_signed`/`military_access_changed` hookup at all. `war_hud.gd` already hooks exactly these three events to its own `_refresh_all()` - `naval_hud.gd` now does the same, closing the gap by matching established precedent rather than inventing a new one. Any transport-specific consequence of a war ending or access changing (a captured port, a revoked access grant) still gets its own dedicated event (`transport_operation_rerouted`, `transport_operation_failed`, etc.) from `TransportSystem.process_day()` on the next tick, already hooked before this packet - this closes the *panel-staleness* gap specifically (the panel not refreshing itself the moment access/war state changes), not a simulation-layer gap.

## Verification

- `tests/naval_transport_workflow_hud_test.gd` (headless `main.tscn` integration): a real embark order driven through the UI, then advanced through the full `embarking -> sailing -> disembarking -> completed` lifecycle via the real scheduler - checking, at each state, the resolved destination name, reserved-capacity display, the persistent pre-embark required/available preview, the route line appearing once sailing, cancellation being offered during embarking *and* disembarking but correctly withheld during sailing, the disembarking cancellation tooltip's real consequence text, and the focus-carried-army button's enabled/disabled state and real `ArmyLayer.set_selected_army()` effect. A second section proves `war_declared`/`peace_signed`/`military_access_changed` each independently trigger a real panel refresh (a fleet added directly to the registry, without calling `_refresh_all()`, becomes visible in `fleet_summary_label` only after the relevant event is emitted).
- Registered in `tools/testing/run_all_tests.py`.
- Targeted regression re-run clean after this packet: `naval_hud_integration_smoke`, `naval_fleet_organisation_hud_test`, `naval_fleet_home_port_hud_test`, `naval_fleet_scuttle_hud_test`, `naval_fleet_summary_hud_test`, `naval_transport_operation_test`, `naval_transport_recovery_test`, `naval_transport_gate_test`, `naval_destructive_edge_gate_test`, `campaign_interface_shell_smoke` (the last one specifically to confirm the `main.tscn` `NavalHUD`/`ArmyLayer` wiring change didn't disturb the shell's own scene graph expectations).

## Deliberately out of scope for this packet

- **Per-route "danger"** - no player-facing risk query exists to build this from; see above.
- **Cross-navigation for fleets carrying more than one simultaneous transport operation** - the focus button is disabled with an explanatory tooltip rather than guessing; a real multi-operation picker would need `transport_label` restructured into a per-row list, a larger change than this bullet's own value justifies alone.
- **Camera panning on focus** - only selection/highlighting is done; this panel has no `camera_controller` reference, unlike `CampaignInterfaceShell`.
